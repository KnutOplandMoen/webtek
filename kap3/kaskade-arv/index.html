Kapittel 3 · Underkapittel 1.6.4 – 1.6.5

Kaskaden og arv

Når flere CSS-regler treffer samme element, må nettleseren velge én. Når en forelder har en stil, må barna bestemme om de skal beholde den. Disse to mekanismene — kaskaden og arv — er hjertet av CSS.

01 · Pensum 1.6.4

Hvorfor trenger vi en kaskade?

I et ekte stilark vil ofte mange regler matche det samme elementet. Du har kanskje en generell p-regel, en regel for klassen .varsel, og en spesifikk regel for #topp. Hvis to av dem prøver å sette color, må nettleseren bestemme seg.

Pensum 1.6.4 sier at kaskaden er settet med regler som avgjør hvem som vinner. Tre faktorer teller, i denne rekkefølgen:

  1. Spesifisitet — mer spesifikk selektor vinner.
  2. Rekkefølge — ved lik spesifisitet vinner regelen som er skrevet sist.
  3. !important — overstyrer (nesten) alt.
02 · Pensum 1.6.4

Fem kilder til stil

Stil kan komme fra fem ulike steder. Pensum lister dem i synkende presedens — øverst slår nederst, gitt at spesifisiteten er lik:

  1. Inline-stil — direkte på elementet via attributtet style="…". Høyest presedens.
  2. Embedded styles — i en <style>-tag inni HTML-dokumentet, vanligvis i <head>.
  3. External style sheet — egen .css-fil koblet inn med <link>.
  4. Brukerdefinerte regler — stilark sluttbrukeren selv har installert i nettleseren (f.eks. for tilgjengelighet).
  5. Browser default — nettleserens standardstil. Brukes når ingen andre nivåer har sagt noe.
Praktisk konsekvens

Hvis du undres over hvorfor en regel ikke slår igjennom: sjekk om det finnes inline-stil eller embedded stil som har høyere presedens. Inline-stil i HTML-en din kan «overdøve» det fineste eksterne stilarket.

MCQ · 1.6.4
Hvilken stil-kilde har høyest presedens i kaskaden?
  • External style sheet
  • Embedded <style>-blokk
  • Inline style="…" på elementet
  • Browser default
Inline-stil ligger nærmest elementet og overstyrer både embedded og external. Browser default har lavest presedens.
03 · Pensum 1.6.4

Spesifisitet — hvor presist peker du?

En selektor som peker mer presist på elementet, vinner over en mindre presis. Pensum gir en grov rangering:

  1. ID-selektor (#topp) — mest spesifikk.
  2. Klasse-selektor (.varsel) — middels.
  3. Type-selektor (p, div) — mindre spesifikk.
  4. Universell selektor (*) — minst spesifikk.

Sammensatte selektorer summerer spesifisiteten. p b er mer spesifikk enn p alene, fordi den krever to ting: en <b> som ligger inni en <p>. 

p          { color: black; }   /* type           */
.varsel    { color: orange; }  /* klasse — vinner over type */
#topp      { color: red; }     /* id  — vinner over klasse  */
MCQ · 1.6.4
Hvilken selektor er mest spesifikk?
  • p
  • .varsel
  • #topp
  • *
ID > klasse > type > universell. ID kan kun forekomme én gang per side, så den peker veldig presist.
MCQ · 1.6.4
Hvorfor vinner p b over p hvis begge treffer en <b> inni en <p>?
  • Fordi p b står først i CSS-filen
  • Fordi b er mer presis enn p
  • Fordi p b er mer spesifikk — den krever to elementer i en bestemt nesting
  • Fordi p alene aldri matcher en <b>
Sammensatte selektorer summerer spesifisitet. Begge regler treffer <b>-elementet (siden p matcher foreldre-p-en, men det er kun p b som er en sammensatt selektor som treffer selve <b>).
04 · Pensum 1.6.4

Siste regel og !important

Hvis to regler er like spesifikke og treffer samme egenskap på samme element, vinner regelen som er skrevet sist — altså lengst ned i CSS-filen (eller siste sammenkoblede stilark). 

.varsel { color: orange; }
.varsel { color: green; }   /* vinner — siste regel */

!important markerer en erklæring som «ekstra viktig», og overstyrer normal spesifisitet. Pensum advarer eksplisitt mot misbruk: hvis to regler begge har !important, må nettleseren bruke vanlige spesifisitetsregler igjen — og kaskaden blir uleselig. 

p          { color: black !important; }
#topp      { color: red; }   /* taper, fordi p har !important */
Pensum-advarsel

«Use sparingly.» !important bør være en siste utvei, gjerne i tredjeparts-stilark du ikke kan endre. I egen kode er det nesten alltid bedre å gjøre selektoren mer spesifikk.

MCQ · 1.6.4
To regler er like spesifikke og setter samme egenskap. Hvilken vinner?
  • Den som står først i filen
  • Den som står sist i filen
  • Den med kortest selektor
  • Den med mest erklæringer
Last rule wins — gitt lik spesifisitet bestemmer rekkefølgen i kildefila.
MCQ · 1.6.4
Gitt p { color: black !important; } og #topp { color: red; } — hvilken farge får <p id="topp">?
  • Rød, fordi ID er mest spesifikk
  • Svart, fordi !important overstyrer spesifisitet
  • Grå, en kombinasjon
  • Standardfarge fra nettleseren
!important går foran spesifisitet. Det er derfor pensum advarer mot misbruk — det «bryter» den vanlige logikken.
05 · Pensum 1.6.5

Arv — hva renner ned til barna?

Når du setter color eller font-family på body, ser du ofte at hele dokumentet får denne fargen eller fonten — uten at du har skrevet en regel for hvert enkelt barn-element. Det er arv i praksis.

Pensum 1.6.5 deler egenskapene i to grupper:

Arves automatisk
color, font-family, font-size, line-height, text-align, visibility — typisk tekstrelaterte egenskaper.
Arves ikke
border, margin, padding, background, width, height — boks-relaterte egenskaper.

Du kan tvinge arv selv med verdien inherit: 

.kort      { border: 2px solid black; }
.kort > *  { border: inherit; }   /* tvinger barna til å arve rammen */
Hvorfor er det slik?

Tankegangen i pensum: tekst-stil bør arves (du vil normalt ikke ha tre forskjellige fonter inni samme avsnitt), mens boks-egenskaper bør ikke arves (ellers ville en border på body dukket opp rundt hvert eneste underelement i hele siden).

MCQ · 1.6.5
Hvilken egenskap arves IKKE automatisk fra forelder til barn?
  • color
  • font-family
  • border
  • line-height
Boks-egenskaper som border, margin og padding arves ikke. Tekst-egenskaper gjør det.
MCQ · 1.6.5
Hvordan tvinger du en egenskap til å arve verdien fra forelder eksplisitt?
  • Med verdien auto
  • Med verdien inherit
  • Med !parent
  • Det går ikke å overstyre arv
inherit er nøkkelordet. auto brukes for layout (f.eks. margin: auto for sentrering), ikke for arv.
06 · Eksamensscenario

Hvilken regel vinner?

HTML
<div id="boks" class="varsel">
  Hei!
</div>
CSS
div          { color: black;  }
.varsel      { color: orange; }
#boks        { color: red;    }
.varsel      { color: green;  }
Tankegangen
  1. Alle fire reglene matcher div-en.
  2. Spesifisitet: #boks (ID) > .varsel (klasse) > div (type).
  3. De to .varsel-reglene konkurrerer kun seg imellom (siste vinner: green) — men begge taper mot #boks.
  4. Konklusjon: teksten blir rød.
MCQ · Scenario
Med samme HTML/CSS som over — hva blir fargen om vi i tillegg legger til .varsel { color: blue !important; }?
  • Rød — ID slår alltid klasse
  • Blå — !important overstyrer normal spesifisitet
  • Grønn — siste regel
  • Svart — type-regel som fallback
!important løfter erklæringen til «vinner», uavhengig av at #boks normalt ville vært mer spesifikk.

MCQ · 1.6.4
Hvilken rekkefølge av faktorer avgjør hvilken regel som vinner?
  • Lengde av selektor → kommentarer → farge
  • Spesifisitet → siste regel → !important
  • Alfabetisk rekkefølge på egenskap
  • Tilfeldig — nettleseren velger
Pensum 1.6.4 lister disse tre faktorene som det som styrer kaskaden.
MCQ · 1.6.5
Hvorfor er det smart å sette font-family på body i stedet for på hvert element?
  • For å gjøre filen større
  • Fordi font-family arves — alle barn av body får samme font automatisk
  • Fordi font-family bare virker på body
  • Fordi nettleseren ignorerer fonter på andre elementer
Tekst-egenskaper arves automatisk. DRY i praksis: skriv det én gang, virker overalt.

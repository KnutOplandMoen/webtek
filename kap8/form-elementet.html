01 · Form-elementet

Hva er form-elementet?

<form> er beholderen som omslutter alle kontrollene i et skjema. Det er et blokkelement — på samme måte som <div> — som starter på en ny linje og fyller bredden. Innholdet er widgetene (tekstfelt, radioknapper, osv.) og en submit-knapp.

Definisjon (forelesning)

Slides forms.md: «Web forms collect information from users.» En bruker fyller ut et skjema og trykker en knapp for å sende informasjonen til serveren — der den behandles av et programmeringsspråk som PHP, C#, Java, JavaScript, Python eller andre.

Dataen trenger ikke nødvendigvis sendes til en tjener — i dette kurset bruker vi ofte JavaScript til å lese verdiene og bruke dem lokalt på klienten. Men selv om innsendingen håndteres lokalt, er det <form>-elementet som grupperer kontrollene og gir dem deres rolle som «data-samlere».

Tomt form-element
<form action="index.html" method="post">
</form>
02 · Attributter

action og method

Form-elementet har to viktige attributter:

action
action="..."

URL-en som data sendes til når brukeren submit-er. Kan være absolutt (https://eksempel.no/register) eller relativ (/register, kontakt.html).

method
method="get" eller method="post"

HTTP-metoden som brukes. GET legger data i URL-en, POST sender data i request-body.

I praksis i dette kurset

I oblig'er og eksamen håndterer vi ofte form-data klientside med JavaScript i stedet for å sende til en server. Du skal likevel kunne forklare hva action og method gjør og velge mellom GET og POST — foreleser har viet flere slides til dette.

03 · GET vs POST

Hvor havner data?

Valget mellom GET og POST handler om hvor dataen ligger, hvor synlig den er, og hva slags handling skjemaet representerer.

GET

Data legges på URL-en etter ? som navn/verdi-par: ?navn=Per&alder=20.

  • Synlig i URL og browser-historikk
  • Kan bookmarkes og deles
  • Kan caches av nettleser/proxy
  • Maks ~2048 tegn (IE-grense)
  • Brukes til oppslag, søk, filtre
POST

Data sendes som en separat melding i HTTP-bodyen — ikke synlig i URL-en.

  • Ikke synlig i URL eller historikk
  • Kan ikke bookmarkes eller deles via lenke
  • Caches ikke som standard
  • Ingen praktisk grense på datamengde
  • Brukes til endringer, passord, filopplastning
Tommelfingerregel

Hvis skjemaet endrer noe på tjeneren (registrerer bruker, oppretter post, sletter noe) — bruk POST. Hvis det bare spør om noe (søk, filter) — bruk GET. Sensitive data (passord, betaling) skal alltid sendes med POST.

Sjekk · Lett
Et skjema lar brukeren søke i en produktkatalog. Hvilken metode passer best?
  • A GET — søk er en safe oppslagsoperasjon, brukeren kan bookmarke resultatet
  • B POST — alltid sikrest
  • C PUT — søk er en oppdatering
  • D Begge gir samme resultat
Riktig: A. Søk er idempotent og safe — naturlig for GET. Bonus: brukeren kan dele eller bookmarke søkeresultatet.
Sjekk · Middels
Hva skjer hvis du sender et passord med method="get"?
  • A Det blir kryptert automatisk
  • B Passordet havner i URL-en — synlig i historikk, cache og serverlogger
  • C Nettleseren stopper innsendingen
  • D Passordet konverteres til POST automatisk
Riktig: B. GET legger alle felt i URL-en. Passord skal alltid sendes med POST.
04 · Vanlige feller

Vanlige feller

Felle 1 — GET med passord

method="get" legger alle felt i URL-en. Passord havner da i nettleserhistorikk, server-logger og kan deles via lenker uten at brukeren forstår det.

Felle 2 — glemt action eller method

Foreleser viser at hvert form-element burde ha action (URL data sendes til) og typisk method + id. Mangler du action, går submit til selve siden — sjelden ønsket.

Felle 3 — file-upload uten POST

type="file" krever method="post". Slides sier det rett ut: «Type="file" you can use only post method.»

05 · Test deg selv

Multiple choice

1 · Lett
Hvilket attributt på <form> bestemmer URL-en data sendes til?
  • A method
  • B action
  • C target
  • D href
Riktig: B — action. method er HTTP-metoden, href brukes på <a>, og target styrer hvilket vindu lenken åpnes i.
2 · Lett
<form> er hva slags element?
  • A Inline-element
  • B Blokkelement
  • C Void-element
  • D Self-closing element
Riktig: B. <form> er et blokkelement, akkurat som <div> — starter på ny linje og fyller bredden.
3 · Middels
Hvor havner data ved method="post"?
  • A I URL-en etter ?
  • B I HTTP-bodyen, som en separat melding
  • C I en cookie
  • D I localStorage
Riktig: B. POST sender data i request-bodyen — ikke synlig i URL-en, ikke begrenset av URL-lengde, ikke cachet som standard.
4 · Middels
Når GET legger data i URL-en som ?navn=Per&alder=20 — hva kaller foreleseren disse parene?
  • A Cookies
  • B Navn/verdi-par (name/value pairs)
  • C Headers
  • D Query-strings, ikke noe annet
Riktig: B. Slides forms.md: «information is sent from the browser to the server using name/value pairs».
5 · Middels
Hva må method være for et skjema som lar brukeren laste opp en fil?
  • A GET
  • B POST
  • C Begge fungerer like godt
  • D UPLOAD
Riktig: B. Slides: «Type="file" you can use only post method.»
6 · Vanskelig
Hvilken av disse er typisk en god grunn til å bruke POST framfor GET?
  • A Du vil at brukeren skal kunne bookmark-e resultatet
  • B Skjemaet er et oppslag eller søk
  • C Skjemaet endrer tilstand på serveren (betaling, registrering, sletting) eller har sensitive data
  • D Du vil at data skal cache-s i nettleseren
Riktig: C. Slides: «Use POST if: Interaction is more like an order, changes state of the resource, or the user will be held accountable.»

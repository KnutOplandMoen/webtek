01 · Pensum 1.9.1

Hva er form-elementet?

<form> er beholderen som omslutter alle kontrollene i et skjema. Det er et blokkelement — på samme måte som <div> — som starter på en ny linje og fyller bredden. Innholdet er widgetene (tekstfelt, radioknapper, osv.) og en submit-knapp.

Definisjon (pensum 1.9)

«Forms allow users to send data to a server through a web site, e.g. sign up for a news letter, register an appointment at the doctors office or signing up for an account.»

Dataen trenger ikke nødvendigvis sendes til en tjener — i dette kurset bruker vi ofte JavaScript til å lese verdiene og bruke dem lokalt på klienten. Men selv om innsendingen håndteres lokalt, er det <form>-elementet som grupperer kontrollene og gir dem deres rolle som «data-samlere».

Tomt form-element
<form action="index.html" method="post">
</form>
02 · Attributter

action og method

Form-elementet har to viktige attributter:

action
action="..."

URL-en som data sendes til når brukeren submit-er. Kan være absolutt (https://eksempel.no/register) eller relativ (/register, kontakt.html).

method
method="get" eller method="post"

HTTP-metoden som brukes. GET legger data i URL-en, POST sender data i request-body.

Pensum-nyans

Pensum 1.9.1 nevner at action og method ikke brukes aktivt i dette kurset (vi håndterer ofte data klientside med JavaScript), men de er fortsatt en del av eksamenspensum. Du må kunne forklare hva de gjør.

03 · GET vs POST

Hvor havner data?

Valget mellom GET og POST handler om hvor dataen ligger, hvor synlig den er, og hva slags handling skjemaet representerer.

GET

Data legges på URL-en etter ? som navn/verdi-par: ?navn=Per&alder=20.

  • Synlig i URL og browser-historikk
  • Kan bookmarkes og deles
  • Kan caches av nettleser/proxy
  • Maks ~2048 tegn (IE-grense)
  • Brukes til oppslag, søk, filtre
POST

Data sendes som en separat melding i HTTP-bodyen — ikke synlig i URL-en.

  • Ikke synlig i URL eller historikk
  • Kan ikke bookmarkes eller deles via lenke
  • Caches ikke som standard
  • Ingen praktisk grense på datamengde
  • Brukes til endringer, passord, filopplastning
Tommelfingerregel

Hvis skjemaet endrer noe på tjeneren (registrerer bruker, oppretter post, sletter noe) — bruk POST. Hvis det bare spør om noe (søk, filter) — bruk GET. Sensitive data (passord, betaling) skal alltid sendes med POST.

Sjekk · Lett
Et skjema lar brukeren søke i en produktkatalog. Hvilken metode passer best?
  • A GET — søk er en safe oppslagsoperasjon, brukeren kan bookmarke resultatet
  • B POST — alltid sikrest
  • C PUT — søk er en oppdatering
  • D Begge gir samme resultat
Riktig: A. Søk er idempotent og safe — naturlig for GET. Bonus: brukeren kan dele eller bookmarke søkeresultatet.
Sjekk · Middels
Hva skjer hvis du sender et passord med method="get"?
  • A Det blir kryptert automatisk
  • B Passordet havner i URL-en — synlig i historikk, cache og serverlogger
  • C Nettleseren stopper innsendingen
  • D Passordet konverteres til POST automatisk
Riktig: B. GET legger alle felt i URL-en. Passord skal alltid sendes med POST.
04 · Pensum 1.9.4

Designe et skjema

Pensum 1.9.4 understreker ett mål: brukeren skal fullføre skjemaet raskt og uten forvirring. Det sparer tid, og det reduserer mengden «feilaktig» data som sendes inn.

HTML inni form

Inne i <form> kan du bruke alt vanlig HTML: <h1>–<h6> som overskrifter, <p>/<br> for tekstbryting, <b>/<i> for fremheving. Skjemaer er ikke en isolert sandkasse.

Fire attributter for brukeropplevelse

A · autofocus
<input ... autofocus>

Markøren går automatisk til feltet når siden lastes. Brukes på det viktigste feltet på siden — typisk søkefeltet på en søkeside (f.eks. google.com).

Bruk ikke på et lite kontaktskjema nederst på en artikkel — det stjeler fokus fra det brukeren egentlig kom for å lese.

Søkefelt med autofokus
<form>
  <label for="search">Søk:
    <input type="text" name="query" id="search" autofocus>
  </label>
</form>
B · disabled
<input ... disabled>

Feltet er ikke aktivt — kan ikke fylles ut, ikke fokuseres, sendes ikke ved submit. Brukes når feltet ikke skal kunne endres før noe annet er gjort. Må fjernes med JavaScript for å aktivere senere.

Areal beregnes automatisk
<form>
  <label for="length">Lengde:
    <input type="number" name="length" id="length">
  </label>
  <label for="width">Bredde:
    <input type="number" name="width" id="width">
  </label>
  <label for="area">Areal:
    <input name="area" id="area" disabled>
  </label>
</form>
C · value
<input ... value="...">

Forhåndsutfylt verdi. Kan hjelpe brukeren (f.eks. by basert på geolokasjon), men brukere overser ofte ferdigutfylte verdier. Bruk med omhu.

Forhåndsutfylt by
<form>
  <label for="city">By:
    <input type="text" name="city" id="city" value="Trondheim">
  </label>
</form>
D · tabindex
<input ... tabindex="N">

Styrer rekkefølgen tab-tasten flytter fokus i. Standard er rekkefølgen elementene står i HTML-kilden.

  • Negativ verdi — fokuserbar, men ikke nåbar via tab.
  • 0 — fokuserbar, naturlig posisjon (HTML-rekkefølge).
  • Positiv verdi — manuell rekkefølge: tab følger økende tabindex.

Elementer med tabindex="0" eller uten attributtet kommer etter elementer med positiv tabindex.

Manuell tab-rekkefølge
<form>
  <label for="name">Fullt navn:
    <input type="text" id="name" name="name" tabindex="1">
  </label>
  <label for="email">E-post:
    <input type="email" id="email" name="email" tabindex="2">
  </label>
</form>
Sjekk · Middels
Hva betyr tabindex="0"?
  • A Elementet kan ikke fokuseres
  • B Elementet er fokuserbart og inngår i tab-rekkefølgen i sin naturlige posisjon (HTML-rekkefølge)
  • C Elementet får alltid fokus først
  • D Elementet hopper over tab-rekkefølgen
Riktig: B. 0 = fokuserbar i naturlig posisjon. Negativ = utenfor tab-flyt. Positiv = manuell rekkefølge.
Sjekk · Lett
På hvilken type side passer autofocus dårlig?
  • A Søkemotor (google.com)
  • B Innloggingsside
  • C Nyhetsartikkel med kontaktskjema nederst
  • D Registreringsskjema som er sidens hovedinnhold
Riktig: C. Autofocus stjeler fokus — uønsket når brukeren kom for å lese artikkelen, ikke for å fylle ut skjemaet.
05 · Vanlige feller

Vanlige feller

Felle 1 — GET med passord

method="get" legger alle felt i URL-en. Passord havner da i nettleserhistorikk, server-logger og kan deles via lenker uten at brukeren forstår det.

Felle 2 — disabled betyr ikke «valgfri»

En disabled input sendes ikke ved submit og kan ikke endres av brukeren. Hvis du vil at brukeren skal kunne velge å fylle ut feltet, ikke bruk disabled.

Felle 3 — value som «hint»

Forhåndsutfylte verdier (value) blir lett oversett av brukere. Hvis du vil foreslå et format, bruk placeholder i stedet.

Felle 4 — autofocus på flere felt

Bare ett felt på siden kan ha fokus. Hvis du setter autofocus på flere, er det siste som vinner — og det er sjelden den du forventet. Begrens til ett.

06 · Test deg selv

Multiple choice

1 · Lett
Hvilket attributt på <form> bestemmer URL-en data sendes til?
  • A method
  • B action
  • C target
  • D href
Riktig: B — action. method er HTTP-metoden, href brukes på <a>, og target styrer hvilket vindu lenken åpnes i.
2 · Lett
<form> er hva slags element?
  • A Inline-element
  • B Blokkelement
  • C Void-element
  • D Self-closing element
Riktig: B. <form> er et blokkelement, akkurat som <div> — starter på ny linje og fyller bredden.
3 · Middels
Hvor havner data ved method="post"?
  • A I URL-en etter ?
  • B I HTTP-bodyen, som en separat melding
  • C I en cookie
  • D I localStorage
Riktig: B. POST sender data i request-bodyen — ikke synlig i URL-en, ikke begrenset av URL-lengde, ikke cachet som standard.
4 · Middels
Hvilket attributt vil du bruke på det viktigste søkefeltet på en søkemotor?
  • A disabled
  • B autofocus
  • C value
  • D tabindex="-1"
Riktig: B. autofocus setter markøren rett i feltet ved sidelast — passer perfekt på sider hvor skjemaet er hovedinnholdet.
5 · Vanskelig
Hvordan fjerner du disabled-attributtet senere så feltet kan fylles ut?
  • A Det er ikke mulig — feltet blir disabled for alltid
  • B Bruk en CSS-regel som overstyrer disabled
  • C Bruk JavaScript til å fjerne attributtet fra elementet
  • D Sett tabindex="0" for å overstyre det
Riktig: C. Pensum 1.9.4 sier eksplisitt at disabled må fjernes med JavaScript når feltet skal aktiveres.
6 · Vanskelig
Du har tre felt med tabindex="3", tabindex="1" og tabindex="2" i den rekkefølgen i HTML-en. I hvilken rekkefølge går tab?
  • A Den rekkefølgen de står i HTML-en (3 → 1 → 2)
  • B Stigende tabindex: 1 → 2 → 3
  • C Synkende tabindex: 3 → 2 → 1
  • D Tilfeldig — nettleseren bestemmer
Riktig: B. Positive tabindex-verdier følges i stigende numerisk rekkefølge, uavhengig av HTML-rekkefølgen.

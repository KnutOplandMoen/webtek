Validering — DTD & XSD

Well-formed handler om syntaks. Valid betyr at dokumentet i tillegg følger en grammatikk — DTD eller XSD — som beskriver hvilke elementer som er lov og hvordan de kan kombineres.

Pensum: 1.14 Forelesning 13
01 · Hvorfor validere?

Well-formed vs valid

Et well-formed dokument følger XML-syntaksen. Men ingenting hindrer deg i å lage <person> uten name, eller å plassere <height> direkte under root. Hvis dataene skal sendes mellom systemer trenger begge sider å vite nøyaktig hvilken struktur som er gyldig.

En grammatikk beskriver dette presist. XML har to slike språk: DTD (eldre, kompakt) og XSD (moderne, mer kraftig).

Well-formed
Følger XML-syntaks: alt lukket, riktig nøstet, ett root, samme case.
Valid
Well-formed og følger en DTD eller XSD som beskriver lovlig struktur.
02 · DTD

Document Type Definition

DTD er det eldste validerings-språket for XML. Det plasseres i prologen — etter XML-deklarasjonen og før selve dokumentet — og bruker en kompakt egen-syntaks (ikke XML).

Inline DTD
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE person [
  <!ELEMENT person (name, height, age)>
  <!ELEMENT name (#PCDATA)>
  <!ELEMENT height (#PCDATA)>
  <!ELEMENT age (#PCDATA)>
]>
<person>
  <name>Name Nameson</name>
  <height>173</height>
  <age>23</age>
</person>

Hver linje i DOCTYPE-blokken sier noe konkret:

<!DOCTYPE person [ … ]>
Erklærer at root er person og åpner DTD-blokken.
<!ELEMENT person (name, height, age)>
person må inneholde nøyaktig disse tre barna, i denne rekkefølgen.
<!ELEMENT name (#PCDATA)>
name inneholder tekst som parses som XML.
03 · PCDATA og CDATA

To slags tekst

PCDATA
«Parsed Character Data» — tekst som parseren tolker som XML. Markup inni leses, så < og & må ha entiteter.
CDATA
«Character Data» — tekst som ikke parses. Alt regnes som ren tekst, så markup blir ignorert. Bra for kodeeksempler i XML.
CDATA-seksjon
<script>
  <![CDATA[
    if (a < b && b < c) { … }
  ]]>
</script>

Inni <![CDATA[ … ]]> kan du skrive < og & rått — parseren rører dem ikke. Nyttig når du skal lagre kode, regulære uttrykk eller annet som ellers ville krevd en haug entity references.

04 · XSD

XML Schema Definition

XSD er et nyere alternativ til DTD. Det viktigste skillet: XSD er selv skrevet i XML-syntaks. Det betyr at samme parser leser både skjemaet og dokumentet — og at XSD kan utvides og valideres på samme måte som annen XML.

person.xsd (utdrag)
<xs:schema xmlns:xs="http://www.w3.org/2001/XMLSchema">
  <xs:element name="person">
    <xs:complexType>
      <xs:sequence>
        <xs:element name="name" type="xs:string"/>
        <xs:element name="height" type="xs:int"/>
        <xs:element name="age" type="xs:int"/>
      </xs:sequence>
    </xs:complexType>
  </xs:element>
</xs:schema>

XSD støtter rikere typer enn DTD — du kan kreve at height faktisk er et heltall, ikke bare «tekst». DTD kan ikke uttrykke det.

05 · XSD-typer

Tre typer du må kunne

Simple type
Element med kun tekstinnhold. Pre-definerte: xs:int, xs:string, xs:boolean, xs:date, xs:decimal …
Complex type
Container for andre elementer. Bygger hierarki — f.eks. person som inneholder name, height, age.
Global type
Definisjon på topp-nivå som gjenbrukes flere steder. Endring (f.eks. å legge til postcode) gjøres bare ett sted.
Hvorfor global type?

Sett at både customer og employee har en adresse. Definer addressType én gang som global complex type, og refererer type="addressType" begge steder. Skal du endre adresse-strukturen senere, gjør du det ett sted.

06 · DTD vs XSD

Når hva?

DTD
Kompakt, eldre, egen syntaks. Kan ikke uttrykke datatyper — alt er tekst. Inline i prologen eller ekstern fil.
XSD
Mer ordrik, men selv XML. Støtter datatyper (int, date …), namespaces, og gjenbruk. Lever som egen .xsd-fil.

I praksis brukes XSD oftest i nye systemer. DTD lever videre i eldre standarder (f.eks. en del HTML- og dokumentformat-arv).

07 · Test deg selv

Multiple choice

1 · Lett
Hva står DTD for?
  • Data Transfer Description
  • Document Type Definition
  • Default Tag Definition
  • Direct Type Description
DTD = Document Type Definition.
2 · Lett
Hva står XSD for?
  • XML Standard Document
  • Extended Schema Description
  • XML Schema Definition
  • XML Strict Data
XSD = XML Schema Definition.
3 · Lett
Hva er forskjellen på well-formed og valid?
  • Synonymer — samme ting
  • Well-formed = riktig syntaks. Valid = well-formed og følger en DTD/XSD
  • Valid betyr at dokumentet er stylet med CSS
  • Well-formed gjelder bare for XSD
Pensum 1.14: well-formed handler om syntaks; valid krever i tillegg at dokumentet følger en grammatikk.
4 · Middels
Hvor i dokumentet plasseres en inline DTD?
  • Etter root-elementet
  • I prologen, etter XML-deklarasjonen og før dokumentets innhold
  • Inni root-elementet
  • I en egen <dtd>-tag
DTD ligger i prologen, mellom <?xml … ?> og selve elementene.
5 · Middels
Hva betyr PCDATA?
  • Pre-Compiled Data Attribute
  • Parsed Character Data — tekst som parses som XML
  • Public Character Data
  • Plain Char Data — som CDATA
Pensum 1.14: PCDATA er tekst som parseren tolker som markup. Trenger entity references for < og &.
6 · Middels
Hva er hovedforskjellen mellom PCDATA og CDATA?
  • PCDATA parses som XML; CDATA behandles som ren tekst og parses ikke
  • PCDATA er tekst, CDATA er tall
  • CDATA brukes bare i XSD
  • Ingen forskjell — bare ulike navn
Inni CDATA kan du ha rå < og & uten entiteter — parseren rører ikke innholdet.
7 · Middels
Hva er en simple type i XSD?
  • Et element med ett enkelt barn
  • Et element med kun tekstinnhold, f.eks. xs:int eller xs:string
  • En type som ikke kan gjenbrukes
  • En forenklet DTD-erklæring
Simple type = bare tekst. Pre-definerte typer inkluderer xs:int, xs:boolean, xs:string, xs:date.
8 · Middels
Hva er en complex type?
  • En type som er vanskelig å forstå
  • En container for andre elementer — bygger hierarki
  • En type med flere encoding-alternativer
  • En type som krever ekstern DTD
Complex type pakker andre elementer (simple eller complex) til et hierarki — f.eks. person som har name, height, age.
9 · Vanskelig
Hvorfor velge XSD framfor DTD i et nytt prosjekt?
  • XSD er kortere å skrive
  • XSD støtter datatyper (int, date …), er selv XML, og kan gjenbrukes med global types
  • DTD er ikke støttet i moderne nettlesere
  • XSD krever ikke prolog
DTD kan bare beskrive struktur, ikke datatyper. XSD gir både typer, gjenbruk og leses av samme parser som dokumentet.
10 · Vanskelig
Hvilken DTD-linje krever at person har name, height, age i den rekkefølgen?
  • <!ELEMENT person [name | height | age]>
  • <!ELEMENT person (name, height, age)>
  • <!ELEMENT person {name, height, age}>
  • <!ELEMENT person = name + height + age>
DTD bruker komma-separert liste i parentes for sekvens. | ville betydd «én av».
11 · Vanskelig
Du skal lagre et JS-snippet med if (a < b) inni XML uten å bruke &lt;. Hva gjør du?
  • Bytter til JSON
  • Setter standalone="yes"
  • Pakker innholdet i en CDATA-seksjon: <![CDATA[ if (a < b) ]]>
  • Bruker en simple type
CDATA er nettopp for slike tilfeller — innholdet parses ikke, så markup-tegn kan stå rått.

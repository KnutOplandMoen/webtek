01 · Helhetsbilde

Hvorfor XML?

XML — Extensible Markup Language — er et markup-språk som ligner på HTML. Forskjellen er hva det brukes til: HTML beskriver en side i en nettleser, mens XML er et generelt markup-språk der du selv definerer taggene. Hovedformålet er å dele data mellom ulike systemer, på samme måte som JSON.

Der HTML har en fast vokabular (<p>, <h1>, <table>), står du fritt i XML til å bruke <person>, <height> eller hva enn dataene dine handler om. Til gjengjeld må XML være well-formed: riktig nøstet, korrekt lukket, og case-sensitivt — ellers godtar parseren det ikke.

Kilde: pensum 1.14 (Extensible Markup Language).

02 · Mental modell

Som HTML, men strengere

Tenk på XML som HTML uten det innebygde vokabularet, og uten slingringsmonn. To regler holder mesteparten av syntaksen sammen:

  1. Ett root-element som omslutter alt annet.
  2. Alt skal lukkes i samme rekkefølge det åpnes, med samme bokstav-case.

Et XML-dokument er en trestruktur: et root-element med barn, som igjen kan ha barn, helt ned til tekst-noder. Strukturen kan valideres mot en DTD eller XSD som beskriver hvilke tagger som er lovlige og hvordan de kan kombineres.

03 · Reisen

Skriv et person-dokument

Bygg dokumentet stegvis — slik kommer alle delene fra pensum 1.14 fram.

  1. Start med XML-deklarasjonen (prologen). Den er valgfri, men hvis den brukes må den stå først: 
    <?xml version="1.0" encoding="UTF-8" standalone="yes"?>
    Versjon, encoding (UTF-8 er default) og standalone (om dokumentet står på egne bein, eller henter informasjon fra eksterne kilder).
  2. Legg til root-elementet. Hvert XML-dokument må ha nøyaktig ett root-element som omslutter alt annet: 
    <persons>
  …
</persons>
  3. Plasser barne-elementer inni. Tagg-navnene velger du selv — de skal beskrive dataene: 
    <persons>
  <person>
    <name>Name Nameson</name>
    <height>173</height>
    <age>23</age>
  </person>
</persons>
  4. Sjekk at alt er well-formed. Lukket i riktig rekkefølge? Samme case på åpne- og lukke-tag? Hvis ikke parses ikke dokumentet.
  5. Erstatt spesialtegn med entity references. Skal dataene inneholde < eller &, bruk &lt; og &amp; — ellers tror parseren det er markup.
  6. Valider mot en DTD eller XSD hvis du vil garantere at strukturen er korrekt — ikke bare well-formed, men valid.

08 · Vanlige feller

Vanlige feller

Pass på
  1. Glemmer lukke-tag. XML har ingen «void»-elementer — alt skal lukkes. Gjør du ikke det er ikke dokumentet well-formed.
  2. Mismatch på case. <Name>…</name> er en feil. XML er case-sensitivt; HTML er ikke.
  3. Feil nesting. <a><b>…</a></b> bryter struktur — barnet må lukkes før forelderen.
  4. Mer enn ett root-element. Et XML-dokument har nøyaktig ett. Hvis du har flere "person", pakk dem i en <persons>.
  5. Bruker reserverte tegn rått. Skriver du < i tekst tror parseren at en ny tag begynner. Bruk &lt;.
  6. Tror HTML er XML. HTML godtar manglende lukke-tag og blanding av case; XML gjør det ikke.
  7. Forveksler well-formed og valid. Well-formed = riktig syntaks. Valid = riktig og følger DTD/XSD.
  8. Plasserer prologen feil. XML-deklarasjonen må stå først i dokumentet hvis den finnes — ingen tegn før den.
09 · Eksamensscenario

Skriv et XML-dokument for noen data

Oppgave

«Skriv et well-formed XML-dokument som beskriver tre personer. Hver person har et navn, høyde og alder. Inkluder en korrekt XML-deklarasjon.»

Eksempelsvar
Slik bygger du opp svaret stegvis — sjekk fasit etterpå.
<?xml version="1.0" encoding="UTF-8" standalone="yes"?>
<persons>
  <person>
    <name>Name Nameson</name>
    <height>173</height>
    <age>23</age>
  </person>
  <person>
    <name>Another Nameson</name>
    <height>181</height>
    <age>25</age>
  </person>
  <person>
    <name>Third Nameson</name>
    <height>168</height>
    <age>30</age>
  </person>
</persons>

Sensor leter etter: prolog først, ett root-element (persons), riktig nesting og lukking, samme case på alle åpne/lukke-tag.

Variant

«Legg til en DTD som krever at hver person inneholder nøyaktig name, height og age i den rekkefølgen.»

Forventet løsning følger mønsteret <!DOCTYPE persons [ <!ELEMENT person (name, height, age)> <!ELEMENT name (#PCDATA)> … ]> plassert etter prologen.

10 · Test deg selv

Helhetsspørsmål

Klikk «Se svar» når du har tenkt gjennom spørsmålet selv.

01 · Lett
Hva står forkortelsen XML for, og hva er hovedformålet med språket?
Extensible Markup Language. Hovedformålet er å dele data mellom ulike systemer — på samme måte som JSON.
02 · Lett
Hvilken viktig forskjell finnes mellom HTML og XML når det gjelder tag-navn?
HTML har et fast sett med tag-navn. I XML definerer du tag-navnene selv — språket er «extensible».
03 · Lett
Skriv en gyldig XML-deklarasjon med versjon 1.0 og UTF-8-encoding.
<?xml version="1.0" encoding="UTF-8"?>
04 · Lett
Hvor i dokumentet må XML-deklarasjonen stå?
Først — før alt annet. Deklarasjonen er valgfri, men hvis den brukes må den være det første i dokumentet.
05 · Middels
Hva betyr det at et XML-dokument er well-formed?
Det følger XML-syntaksen: alle tagger er korrekt lukket, riktig nøstet, det finnes nøyaktig ett root-element, og åpne- og lukke-tag har samme case.
06 · Middels
Hva er forskjellen på «well-formed» og «valid»?
Well-formed handler om syntaks. Valid betyr at dokumentet i tillegg følger en DTD eller XSD som beskriver hvilke tagger som er lov og hvordan de kan kombineres.
07 · Middels
Hvilke fem tegn må erstattes med entity references i XML, og hvilke entiteter brukes?
< → &lt;
> → &gt;
& → &amp;
' → &apos;
" → &quot;
08 · Middels
Vurder om denne snutten er well-formed: <person><name>Per</person></name>
Nei. name er åpnet inni person men lukkes etter person. Reglene for nesting krever at barnet lukkes før forelderen.
09 · Middels
Vurder om denne snutten er well-formed: <Person>Per</person>
Nei. XML er case-sensitivt. <Person> og <person> regnes som ulike tagger.
10 · Middels
Hvor mange root-elementer kan et XML-dokument ha?
Nøyaktig ett. Hvis du har flere objekter på topp-nivå må du pakke dem i et felles root-element.
11 · Middels
Skriv en XML-kommentar som forklarer hva neste element representerer.
<!-- Personen er kursleder -->
12 · Middels
Oversett denne JSON-en til XML: 
{ "person": { "name": "Name Nameson", "height": 173, "age": 23 } }
<person>
  <name>Name Nameson</name>
  <height>173</height>
  <age>23</age>
</person>
13 · Vanskelig
Hva er forskjellen mellom PCDATA og CDATA i en DTD?
PCDATA («Parsed Character Data») er tekst som parseren tolker som XML — markup blir lest. CDATA er tekst som ikke parses; alt behandles som ren tekst, så «<» og «&» trenger ingen entiteter.
14 · Vanskelig
Skriv en DTD som krever at person inneholder name, height og age, alle som tekst.
<!DOCTYPE person [
  <!ELEMENT person (name, height, age)>
  <!ELEMENT name (#PCDATA)>
  <!ELEMENT height (#PCDATA)>
  <!ELEMENT age (#PCDATA)>
]>
15 · Vanskelig
Hvilke tre typer finnes i XSD, og hva brukes de til?
Simple type — for tekstinnhold, f.eks. xs:int, xs:boolean, xs:string, xs:date.
Complex type — container for andre elementer; bygger hierarki.
Global type — én definisjon som gjenbrukes på flere steder, slik at endringer bare trengs ett sted.
16 · Vanskelig
Hvordan kobler du et CSS-stilark til et XML-dokument slik at innholdet vises stylet i nettleser?
Sett en stylesheet-instruksjon etter prologen: <?xml-stylesheet type="text/css" href="stylesheet.css"?>. Deretter bruker du vanlig CSS som om dokumentet var HTML.

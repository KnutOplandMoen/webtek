Hvilken protokoll hører til applikasjonslaget (Application layer) i TCP/IP-modellen?

  • A Ethernet
  • B IP
  • C SMTP
  • D TCP
Vis fasit
Riktig svar: C

Spørsmål 3 2 poeng Kap. 4 — position:fixed referansepunkt

Hvor plasseres et element med position: fixed; top: 0; left: 0;?

  • A Øverst venstre i nærmeste forelder.
  • B Øverst venstre i nettleserens viewport (synlig vindu).
  • C Øverst venstre i hele dokumentet (kan kreve scrolling).
  • D Avhengig av nettleserens innstillinger.
Vis fasit
Riktig svar: B

position: fixed låser elementet til viewport — det synlige nettleser-vinduet. Elementet blir værende på samme sted selv når man scroller. Brukes typisk for "sticky" navmenyer, chat-knapper, eller cookie-bannere.

Hvorfor de andre er feil: A beskriver position: absolute (plassert relativt nærmeste posisjonerte forelder). C er nær absolute uten posisjonert forelder (defaulter til html-elementet). D er feil — fixed er deterministisk per spec.

Spørsmål 4 3 poeng Kap. 5 — flex-direction column

Når en flex-container har flex-direction: column, hvilken egenskap styrer den horisontale plasseringen av barna?

  • A justify-content
  • B align-items
  • C flex-wrap
  • D align-content
Vis fasit
Riktig svar: B

I column-layout snur aksene: hovedaksen (main axis) er vertikal, og tverr-aksen (cross axis) er horisontal. align-items styrer alltid tverr-aksen — som her er horisontal.

Mnemonisk: justify = main axis (avhengig av flex-direction); align = cross axis. I row: justify=horisontal, align=vertikal. I column: justify=vertikal, align=horisontal. Aksene snur, ikke betydningen.

Hvorfor de andre er feil: A styrer hovedaksen, som er vertikal her. C styrer line-wrap-oppførsel. D påvirker plassering av linjer (når flex-wrap: wrap er aktivert), ikke individuelle elementer.

Spørsmål 5 2 poeng Kap. 5 — grid-template-columns

CSS-regelen grid-template-columns: 1fr 2fr 1fr; på en grid-container gjør hva?

  • A Lager tre kolonner der midten er halvparten av bredden, og sidene deler resten likt.
  • B Lager tre kolonner der midten er dobbelt så bred som hver av sidekolonnene.
  • C Lager tre rader, ikke kolonner — fr gjelder rader.
  • D Lager fire kolonner (1 + 2 + 1 = 4 enheter total).
Vis fasit
Riktig svar: B

fr = "fraction of available space". Med 1fr 2fr 1fr deles bredden i 4 deler (1+2+1): første og siste kolonne får hver 25 %, midten 50 % — dobbelt så bred.

Hvorfor de andre er feil: A snur regelen. C er feil — grid-template-columns gjelder kolonner; for rader brukes grid-template-rows. D er konfabulering — tre fr-verdier gir tre kolonner.

Kontrast: Hvis du blander fr med faste enheter — f.eks. 200px 1fr 1fr — så får første kolonne 200px, og resten av plassen deles 1:1 mellom de to fr-kolonnene.

Spørsmål 6 2 poeng Kap. 4 — clear: both

Hva er hensikten med clear: both; i klassisk float-basert CSS-layout?

  • A Sletter alle CSS-regler arvet fra foreldre-elementet.
  • B Tvinger elementet til å plasseres under ethvert floatet element foran det — både venstre- og høyre-floats.
  • C Aktiverer transparent bakgrunn med RGBA.
  • D Fjerner alle padding og margin (samme som all: unset).
Vis fasit
Riktig svar: B

clear: both sier "ikke plasser meg ved siden av floats — jeg vil under". clear: left og clear: right klarerer kun den respektive siden. Klassisk brukt på en "clearfix"-div etter en serie floatede elementer for å hindre at forelderen kollapser.

Hvorfor de andre er feil: A, C og D er konfabulering. CSS har ingen "slett alle regler"-mekanisme (all: revert er nærmest), og clear har ingenting å gjøre med bakgrunn eller padding.

Moderne alternativ: Bruk Flexbox eller Grid i stedet for floats — da slipper du clearfix-hacks.

Spørsmål 7 2 poeng Kap. 3 — Box-model: width

Med default box-sizing: content-box: hvis et div har width: 200px; padding: 10px; border: 5px solid;, hvor mange piksler er den totale synlige bredden?

  • A 200 px
  • B 210 px
  • C 220 px
  • D 230 px
Vis fasit
Riktig svar: D

Med content-box gjelder width bare innholdet. Total bredde = width + padding (begge sider) + border (begge sider) = 200 + 10·2 + 5·2 = 200 + 20 + 10 = 230 px.

Margin telles ikke i synlig bredde, men i hvor mye plass elementet "tar opp" i layouten.

Med border-box (moderne praksis): Padding og border inkluderes i de 200 px — så innholdsbredden krymper til 170 px, men total bredde er 200 px. Mange CSS-resets setter * { box-sizing: border-box; } for å gjøre layout-regning mer intuitiv.

Spørsmål 8 3 poeng Kap. 4 — visibility:hidden vs display:none

Hva er hovedforskjellen mellom visibility: hidden og display: none?

  • A Begge er identiske, bare ulike navn.
  • B visibility: hidden tar fortsatt plass i layouten; display: none fjerner elementet fullstendig.
  • C display: none tar fortsatt plass; visibility: hidden fjerner elementet helt.
  • D visibility: hidden gjelder bare tekst; display: none gjelder hele elementet.
Vis fasit
Riktig svar: B

visibility: hidden gjør elementet usynlig, men det opptar fortsatt plass — andre elementer flytter seg ikke nærmere. Det er som "usynlig blekk". display: none fjerner det fra layouten — andre elementer fyller plassen.

Praktiske bruksområder: visibility: hidden for å skjule noe midlertidig uten å forandre layoutet (f.eks. en tooltip som dukker opp i samme posisjon hver gang). display: none for å fjerne helt (f.eks. menyer som ikke skal være der i mobilvisning).

Hvorfor de andre er feil: A er helt feil — semantisk og visuelt forskjellige. C snur regelen. D er konfabulering — begge gjelder hele elementet.

Bonus: Skjermlesere oppfører seg også forskjellig. display: none hopper helt over elementet; visibility: hidden gjør det samme (men opacity: 0 leser fortsatt opp innholdet).

Spørsmål 9 2 poeng Kap. 4 — overflow: hidden

Hva gjør overflow: hidden på et element?

  • A Skjuler elementet hvis det er for stort til å vises.
  • B Kutter innhold som strekker seg utenfor elementets boks, uten å vise scrollbars.
  • C Skjuler bare bakgrunnsbilde, men ikke tekst.
  • D Legger til scrollbars hvis innholdet er for stort.
Vis fasit
Riktig svar: B

overflow: hidden klipper alt innhold som strekker seg utenfor element-boksen (basert på width og height). Ingen scroll-mulighet — innholdet er bare borte fra synet.

Andre verdier: visible (default — innhold renner ut), scroll (alltid scrollbars), auto (scrollbars kun ved behov).

Bonusbruk: overflow: hidden på en forelder etablerer også et nytt "block formatting context" — som har den effekten å inneholde floats inni. Klassisk clearfix-alternativ.

Spørsmål 10 3 poeng Kap. 5 — margin: 0 auto

Hvilken CSS-regel sentrerer en block-level boks med fast bredde horisontalt i forelderen?

  • A text-align: center;
  • B margin: 0 auto;
  • C display: center;
  • D position: center;
Vis fasit
Riktig svar: B

margin: 0 auto = "0 top/bottom, auto left/right". auto på horisontale marginer ber nettleseren fordele restplassen likt — derfor sentreres elementet. Krever at elementet har en fast (eller maks) bredde, ellers fyller det hele forelderen.

Spørsmål 11 4 poeng Kap. 9 — Spread og Array.from
Ikke i pensum Spread-operatoren ... og Array.from er ikke gjennomgått i forelesningene. For å konkatenere arrays bruk a.concat([4, 5]) (i pensum). Hopp over hvis du øver til eksamen.

Hva skriver dette ut?

let a = [1, 2, 3];
let b = [...a, 4, 5];
console.log(b.length);
Vis fasit
Riktig svar: 5

Mekanisme: Spread-operatoren ...a "pakker ut" elementene i a én og én. Så [...a, 4, 5] blir [1, 2, 3, 4, 5] — fem elementer.

Kontrast — hva spread ikke gjør: [a, 4, 5] uten spread ville gitt et array på tre elementer: [[1,2,3], 4, 5] — der første element er hele arrayet (nested).

Bruksområde: Spread er den moderne måten å konkatenere arrays. Tidligere måtte man bruke a.concat([4, 5]).

Spørsmål 12 4 poeng Kap. 9 — Array.every

Hva skriver dette ut?

let nums = [2, 4, 6, 7, 10];
console.log(nums.every(n => n % 2 === 0));
Vis fasit
Riktig svar: false

Mekanisme: every() returnerer true bare hvis alle elementer passerer testen. Her: 7 % 2 === 1 (ikke 0), så 7 feiler testen, og every returnerer false.

Kontrast:

  • some() returnerer true hvis minst ett element passerer. nums.some(n => n % 2 === 0) ville vært true (mange partall).
  • filter() returnerer alle som passerer som ny array.
  • find() returnerer første som passerer (eller undefined).

Early exit: every stopper ved første false — den iterer ikke nødvendigvis hele arrayet. some stopper ved første true.

Spørsmål 13 4 poeng Kap. 9 — Array.splice return

Hva skriver dette ut?

let a = [10, 20, 30, 40, 50];
let r = a.splice(1, 2, 99);
console.log(r);
Vis fasit
Riktig svar: [20, 30]

Mekanisme: splice(start, deleteCount, ...newItems) returnerer de fjernede elementene som array — ikke den nye listen.

Her: fra indeks 1, fjern 2 elementer (20 og 30), og sett inn 99. Resultat:

  • a blir [10, 99, 40, 50] (mutert!)
  • r blir [20, 30] (det som ble fjernet)

Vanlig feil: Tro at r blir den nye arrayet (det man tenker etter en "edit"). Det er a som er den nye — splice's return er det fjernede.

Spørsmål 14 3 poeng Kap. 9 — String.repeat med 0

Hva skriver dette ut?

console.log("abc".repeat(0).length);
Vis fasit
Riktig svar: 0

Mekanisme: "abc".repeat(0) returnerer tom streng "" — gjentakelse 0 ganger er ingen ting. Lengden av tom streng er 0.

Edge cases:

  • "abc".repeat(1) → "abc" (length 3)
  • "abc".repeat(3) → "abcabcabc" (length 9)
  • "abc".repeat(-1) → RangeError (negativt antall ikke tillatt)
  • "abc".repeat(0.7) → "" (avrundes ned til 0)

Spørsmål 15 3 poeng Kap. 9 — Boolean av tom array

Hva skriver dette ut?

if ([]) {
  console.log("ja");
} else {
  console.log("nei");
}
Vis fasit
Riktig svar: ja

Mekanisme: En tom array [] er et objekt, og objekter er alltid truthy i JS — uavhengig av hva de inneholder. Derfor går vi inn i if-grenen.

Spørsmål 16 4 poeng Kap. 4 — Position static/relative/absolute/fixed

Forklar de fire CSS-position-verdiene static, relative, absolute og fixed. For hver: hva er referansepunktet for top/left-offsets, og hvordan oppfører elementet seg i normal dokument-flyt?

Vis fasit

Rubrikk (4 poeng — 1 poeng per verdi):

  • static (default): Elementet plasseres i normal dokument-flyt. top/left/right/bottom og z-index ignoreres.
  • relative: Elementet plasseres først i normal flyt, og kan deretter forskyves med top/left. Den opprinnelige plassen i flyten beholdes — andre elementer "vet ikke" at det er forskjøvet. Brukes også ofte for å være referanseramme for absolute-barn.
  • absolute: Tatt ut av flyt. Plasseres relativt til nærmeste posisjonerte forelder (ikke static) — eller <html> hvis ingen finnes. Andre elementer fyller plassen som om det ikke var der.
  • fixed: Tatt ut av flyt. Plasseres relativt til viewport (synlig nettleser-vindu). Blir værende på samme sted ved scroll. Bra for sticky navmenyer.

Bonus: sticky er en hybrid — oppfører seg som relative til en viss scroll-posisjon, så blir den fixed ved nådd terskel.

Spørsmål 17 3 poeng Kap. 1 — POST vs GET (caching og bookmarks)

Forklar hvorfor GET-forespørsler kan caches og bookmarkes, mens POST-forespørsler typisk ikke kan det. Gi tre konkrete argumenter.

Vis fasit

Rubrikk (3 poeng):

  • 1 poeng — GET er en henteoperasjon: GET er ment for å lese data uten å endre noe på serveren. Siden samme GET-forespørsel skal gi samme resultat, kan nettleseren og mellomliggende proxies trygt mellomlagre svaret og levere det igjen uten å spørre serveren på nytt.
  • 1 poeng — POST endrer state: POST brukes til å sende inn data som endrer tilstanden på serveren (f.eks. legge inn en ordre, opprette en bruker). Gjentar man en POST kan man opprette to like rader/ordrer, så nettleseren caches normalt ikke svaret — den må gå til serveren hver gang.
  • 1 poeng — Bookmarks: URL-en for en GET inneholder alle parameterne (etter ?) — det er en komplett "oppskrift" som kan lagres som bookmark og kjøres på nytt. POST-data ligger i request-body, som ikke er en del av URL-en — det er ingenting å bookmarke.

Bonus å nevne: Hvis du prøver å reload en POST-side i nettleseren, får du typisk en advarsel: "Forespørselen kan ikke gjentas — fortsette?" Det er fordi POST ikke skal gjentas implisitt.

Spørsmål 18 3 poeng Kap. 3 — Border-box vs content-box

Forklar forskjellen mellom box-sizing: content-box (default) og box-sizing: border-box. Hva skjer med faktisk bredde hvis du har width: 300px; padding: 20px; border: 2px solid; i begge tilfeller?

Vis fasit

Rubrikk (3 poeng):

  • 1 poeng — content-box: width gjelder kun innholdet. Total synlig bredde = width + 2·padding + 2·border = 300 + 40 + 4 = 344 px.
  • 1 poeng — border-box: width gjelder hele boksen (innhold + padding + border). Innholdsbredden krymper for å gi plass til padding og border: 300 - 40 - 4 = 256 px innhold, men total bredde fortsatt 300 px.
  • 1 poeng — Hvorfor border-box er populært: Layout-beregning blir mer intuitivt. "Jeg vil at boksen skal være 300 px bred" gir 300 px bred. Med content-box må man trekke fra padding/border manuelt for å regne ut innholdsbredden.

Vanlig praksis: Mange CSS-resets bruker * { box-sizing: border-box; } for å gjøre hele dokumentet konsistent.

Spørsmål 19 4 poeng Kap. 5 — Flexbox vs Grid: når bruker man hva?

Du skal designe en layout. Forklar når Flexbox er mer egnet enn Grid, og når Grid er mer egnet enn Flexbox. Gi ett konkret eksempel for hver.

Vis fasit

Rubrikk (4 poeng):

  • 1 poeng — Flexbox er én-dimensjonal: Best for å plassere elementer langs én akse (horisontal eller vertikal). Når antall elementer kan variere og du vil at de fordeler seg dynamisk.
  • 1 poeng — Eksempel på Flexbox-egnethet: En navmeny med varierende antall lenker, en knapprad i en dialog, vertikal sentrering av innhold. Klassiker: "header med logo til venstre og knapper til høyre".
  • 1 poeng — Grid er to-dimensjonal: Best for å plassere elementer i et eksplisitt rutenett med både rader og kolonner. Designet for layouts hvor du planlegger hvor hver del skal være.
  • 1 poeng — Eksempel på Grid-egnethet: Et magasin-layout med header, sidebar, hovedinnhold og footer i eksplisitte celler. Et bildegalleri med faste kort i et rutenett. Et dashboard med widgets.

Bonus: Praksisen er ofte å kombinere — bruk Grid for det overordnede side-layoutet, og Flexbox inne i hver celle for å organisere innholdet på én linje. "Grid for layoutet av siden; Flex for layoutet i komponentene."

Spørsmål 20 3 poeng Kap. 4 — z-index og stacking context

Hva er et stacking context, og hvorfor kan et element med z-index: 9999 noen ganger likevel være "under" et annet element med lavere z-index? Forklar med ett konkret scenario.

Vis fasit

Rubrikk (3 poeng):

  • 1 poeng — Definisjon: Et stacking context er en "gruppe" elementer som stables sammen. z-index sammenlignes kun innenfor samme stacking context. Mellom contexts er det forelderens z-index som bestemmer rekkefølgen, ikke barnas individuelle verdier.
  • 1 poeng — Hva oppretter en context: Root-elementet, alle elementer med position: ... og en numerisk z-index, samt elementer med opacity < 1, transform, filter, eller will-change.
  • 1 poeng — Scenario: Forelder A har z-index: 1; position: relative; et barn av A har z-index: 9999. Forelder B (søsken av A) har z-index: 2; position: relative. Til tross for at barnet i A har z-index 9999, vil B alltid være "over" det — fordi A og B er to forskjellige stacking contexts, og A (z-index 1) er under B (z-index 2). Barnets høye z-index gjelder bare innenfor A's context.

Praktisk konsekvens: Modaler/popups som "skal være over alt" må ofte rendres som direkte barn av <body>, ikke nested inne i en card-komponent — for å unngå å bli fanget i forelderens stacking context.

Spørsmål 21 8 poeng Kap. 5 — Tre-kolonners flex-layout

HTML-en under er gitt. Skriv CSS som lager en tre-kolonners layout med Flexbox:

  1. (2 poeng) <main class="layout"> skal være en flex-container.
  2. (3 poeng) Sidemenyene <aside class="venstre"> og <aside class="hoyre"> skal være fast bredde (200px hver).
  3. (2 poeng) <section class="senter"> skal vokse for å fylle resten av bredden (bruk flex: 1 eller flex-grow: 1).
  4. (1 poeng) Gi de tre kolonnene synlige bakgrunner og litt padding, slik at strukturen er klar i preview.
Vis fasit

Referanseløsning:

.layout {
  display: flex;
  gap: 12px;
}

.venstre, .hoyre {
  flex: 0 0 200px;       /* grow=0, shrink=0, basis=200px */
  background: #e7e3d4;
  padding: 12px;
}

.senter {
  flex: 1;               /* eller flex-grow: 1; */
  background: #faf3df;
  padding: 12px;
}

Kommentarer:

  • flex: 0 0 200px: Shorthand for "grow=0, shrink=0, basis=200px" — låser bredden til 200px og hindrer at den vokser eller krymper. Alternativ: bare width: 200px; flex-shrink: 0;.
  • flex: 1: Shorthand for "grow=1, shrink=1, basis=0%". Tilsvarer "ta opp den resterende plassen".
  • gap: Den moderne måten å sette avstand mellom flex-barn — alternativet er margin på hvert barn, som er tungvint.

Vanlige feil: Bruke width: 200px uten flex-shrink: 0 — Flexbox kan krympe barna under deres bredde-verdi hvis det blir trangt. Bruke justify-content: space-between for å fordele bredden — det fungerer bare hvis de tre boksene har lik bredde. Glemme display: flex på forelderen.

  grid-template-columns: 1fr 2fr;
  grid-template-areas:
    "header header"
    "aside main";
  gap: 12px;
}

.g-head  { grid-area: header; background: #4a2c2a; color: white; padding: 16px; }
.g-side  { grid-area: aside;  background: #e7e3d4; padding: 16px; }
.g-main  { grid-area: main;   background: #faf3df; padding: 16px; }

Kommentarer:

  • grid-template-areas: En "ASCII-art"-måte å definere layout. Hver linje er en rad, og hvert "ord" er et navngitt område som tar én celle. Gjentakelse av samme navn på rad gjør at området spanner over flere kolonner — her header header spanner hele rad 1.
  • grid-area: Kobler et element til et navngitt område. Verdiene må nøyaktig matche navnene fra grid-template-areas (case-sensitive).
  • Hvorfor 1fr 2fr: Sidebar tar 1/3 av bredden, hovedinnhold 2/3. Justerbart etter design — bytt til 200px 1fr for fast sidebar.

Vanlige feil: Glemme anførselstegnene rundt hver rad i grid-template-areas — det er en parser-feil. Forskjellige navn på samme rad mellom CSS og grid-area. Bruke grid-row/grid-column-syntaks (også gyldig, men mye mer skrevet for samme effekt).

Spørsmål 23 8 poeng Kap. 10 — Countdown timer

HTML-en under har et input-felt for antall sekunder, en start-knapp og et display. Skriv JavaScript som:

  1. (2 poeng) Når brukeren klikker «Start», leses antall sekunder fra input.
  2. (3 poeng) Bruk setInterval til å telle ned hver sekund, og oppdater <span id="display"> med gjenværende sekunder.
  3. (2 poeng) Når tellingen når 0, stopp intervallet med clearInterval og vis "Ferdig!" i displayet.
  4. (1 poeng) Håndter ugyldig input (negative tall eller ikke-tall) med en feilmelding.
Vis fasit

Referanseløsning:

const startBtn = document.getElementById("start");
const sekInput = document.getElementById("sek");
const display = document.getElementById("display");

let timerId = null;

startBtn.addEventListener("click", function () {
  // stopp ev. eksisterende timer
  if (timerId !== null) clearInterval(timerId);

  let igjen = parseInt(sekInput.value, 10);
  if (isNaN(igjen) || igjen < 1) {
    display.textContent = "Skriv et positivt heltall";
    return;
  }

  display.textContent = igjen;
  timerId = setInterval(function () {
    igjen--;
    if (igjen <= 0) {
      clearInterval(timerId);
      timerId = null;
      display.textContent = "Ferdig!";
    } else {
      display.textContent = igjen;
    }
  }, 1000);
});

Kommentarer:

  • setInterval: Tar (callback, milliseconds). Returnerer en id som kan brukes med clearInterval(id) for å stoppe.
  • Stopp eksisterende timer: Hvis brukeren klikker Start to ganger, må vi stoppe den første — ellers løper to timere samtidig og tallene blir tullete.
  • Hvorfor sjekke igjen <= 0 inne i callbacken: setInterval venter første gang også 1 sekund — så hvis vi vil vise tallet umiddelbart, må vi gjøre det før setInterval starter.
  • Alternativ: setTimeout-rekursjon: function tick() { ... setTimeout(tick, 1000); } — gir mer kontroll på drift, men er mer kode.

Vanlige feil: Glemme å lagre timerId, så vi kan ikke stoppe. Glemme parseInt så igjen blir en streng og igjen-- gir NaN. Glemme å sjekke om timer allerede kjører — gir doble teller.

Spørsmål 24 7 poeng Kap. 10 — Scroll-to-top knapp

HTML-en under har en lang side med mye innhold og en "scroll-til-toppen"-knapp. Skriv kode som:

  1. (2 poeng) Bruk CSS for å plassere <button id="topp"> position: fixed nederst til høyre, og skjul den i utgangspunktet (display:none).
  2. (3 poeng) Når brukeren har scrollet mer enn 300px nedover, vis knappen. Når de scroller tilbake til toppen, skjul den igjen.
  3. (2 poeng) Når brukeren klikker knappen, scroll smooth til topp av siden.
Vis fasit

Referanseløsning:

/* CSS */
#topp {
  position: fixed;
  bottom: 20px;
  right: 20px;
  padding: 10px 16px;
  background: #5b1f1a;
  color: white;
  border: none;
  border-radius: 6px;
  cursor: pointer;
  display: none;
}

/* JS */
const knapp = document.getElementById("topp");

window.addEventListener("scroll", function () {
  if (window.scrollY > 300) {
    knapp.style.display = "block";
  } else {
    knapp.style.display = "none";
  }
});

knapp.addEventListener("click", function () {
  window.scrollTo({ top: 0, behavior: "smooth" });
});

Kommentarer:

  • position: fixed: Låser knappen til viewport — blir værende synlig under scroll. bottom: 20px; right: 20px; plasserer den 20px fra nederste høyre hjørne.
  • window.scrollY: Gir antall piksler scrollet fra toppen. Alternativer: document.documentElement.scrollTop (mer cross-browser, men mer kode).
  • scrollTo med behavior smooth: Moderne syntaks for animert scroll. Eldre måte: window.scrollTo(0, 0) uten animasjon, eller jQuery's $("body").animate({scrollTop: 0}).

Vanlige feil: Lytte på document i stedet for window for scroll-event — fungerer sjelden. Bruke scrollTop = 0 direkte på window — det funker ikke (må gå via document.documentElement). Glemme å skjule knappen i utgangspunktet — synlig før scroll skjer.

Spørsmål 25 7 poeng Kap. 3+4 — Float + clear

HTML-en under har et bilde og to avsnitt. Skriv CSS som:

  1. (2 poeng) Floater <img> til venstre, slik at første avsnitt flyter rundt det på høyre side.
  2. (2 poeng) Gir bildet 12px margin på høyre og bunnen, slik at tekst ikke klistrer seg fast.
  3. (3 poeng) Andre avsnitt (med class="bunn") skal ikke flyte rundt bildet — det skal plasseres helt under det floatede bildet (bruk clear).
Vis fasit

Referanseløsning:

img {
  float: left;
  margin: 0 12px 12px 0;
}

.bunn {
  clear: left;     /* eller clear: both */
}

Kommentarer:

  • float: left: Tar bildet ut av normal flyt og plasserer det til venstre. Påfølgende tekst flyter rundt på høyre side.
  • margin: Shorthand er top right bottom left. Her: 0 over, 12px høyre (mellom bilde og tekst), 12px under (hvis bildet er høyere enn første avsnitt), 0 venstre.
  • clear: left: Tvinger elementet til å starte under alle floats til venstre. clear: both ville fungert også — tvinger under både venstre- og høyre-floats.
  • Float er klassisk teknologi: Brukes mest historisk og for tekst-flyt rundt bilder. For moderne layouts er Flexbox og Grid bedre — float er bevart i pensum mest for å forstå eldre kode og clear-mekanikken.

Vanlige feil: Glemme clear — andre avsnitt flyter også rundt bildet. Bruke display: block i stedet for clear (gjør ingenting). Sette clear på første avsnitt i stedet for andre.

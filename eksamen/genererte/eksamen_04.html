Spørsmål 2 3 poeng Kap. 1 — URL-anatomi

I URL-en https://www.ntnu.no:8080/it2805/forelesning?dag=mandag#del-3, hva er query string?

  • A www.ntnu.no
  • B /it2805/forelesning
  • C dag=mandag
  • D del-3
Vis fasit
Riktig svar: C

Query string er den delen som kommer etter ?, før et eventuelt #. Den brukes til å sende parametere til serveren.

Anatomien: scheme (https) → host (www.ntnu.no) → port (8080) → path (/it2805/forelesning) → query (dag=mandag) → fragment (del-3).

Hvorfor de andre er feil: A er host. B er path. D er fragment — den delen som kommer etter # og brukes til å hoppe til en ankerseksjon i selve siden (sendes ikke til server).

På hvilket lag i TCP/IP-modellen befinner HTTP seg?

  • A Physical / Link layer
  • B Network / Internet layer
  • C Transport layer
  • D Application layer
Vis fasit
Riktig svar: D

Spørsmål 4 3 poeng Kap. 2 — Semantiske HTML5-tags

Hvilken HTML5-tag er mest egnet for innholdsdelen som inneholder relatert tilleggsinformasjon (f.eks. en sidebar med "Relaterte artikler")?

  • A <section>
  • B <aside>
  • C <footer>
  • D <article>
Vis fasit
Riktig svar: B

<aside> markerer innhold som er tangentielt relatert til hovedinnholdet — typisk en sidemeny, "se også"-boks, eller annonse. Det forteller skjermlesere og søkemotorer at innholdet ikke er kjernen.

Hvorfor de andre er feil: <section> er en tematisk gruppering av relatert innhold med en heading. <footer> er for fotinnhold (copyright, kontakt). <article> er for selvstendig, gjenbrukbart innhold (en blogg-post, en nyhet).

Spørsmål 5 2 poeng Kap. 3 — !important

Gitt følgende CSS:

p { color: blue !important; }
#tittel { color: red; }

Hvilken farge får <p id="tittel">Hei</p>?

  • A Blå — fordi !important overstyrer alt annet.
  • B Rød — fordi id-selektoren har høyere specificity.
  • C Svart — fordi reglene er motstridende og ignoreres.
  • D Først rød, så blå — kaskaden bestemmer rekkefølge.
Vis fasit
Riktig svar: A

!important hever en regel over det vanlige specificity-hierarkiet. Selv om #tittel normalt slår en element-selektor (100 mot 1), er blue !important sterkere.

Hvorfor de andre er feil: B er korrekt regning uten !important. C og D er konfabulering — CSS er deterministisk, ikke "ignorert" eller "begge".

Bonus: Måten å slå !important på er å bruke !important selv og ha høyere specificity (eller skrive senere i kilden).

Spørsmål 6 3 poeng Kap. 4 — position default

Hva er default-verdien for CSS-egenskapen position?

  • A relative
  • B absolute
  • C static
  • D auto
Vis fasit
Riktig svar: C

Alle elementer er position: static som standard. Det betyr at top, right, bottom, left og z-index ignoreres — elementet flyter i normal dokument-flyt.

Konsekvens: Hvis du vil bruke position: absolute på et barn, må forelderen ha position: relative (eller noe annet enn static) — ellers absoluteres barnet til <html>.

Hvorfor de andre er feil: A, B og fixed krever eksplisitt deklarasjon. D er ikke en gyldig verdi for position (men er det for f.eks. margin og width).

Spørsmål 7 2 poeng Kap. 5 — align-items vs justify-content

I en flex-container med flex-direction: row, hvilken egenskap styrer plasseringen langs den vertikale (tverr-)aksen?

  • A justify-content
  • B align-items
  • C flex-wrap
  • D flex-grow
Vis fasit
Riktig svar: B

align-items styrer plassering langs cross axis (tverr-aksen). I row er hovedaksen horisontal, så tverr-aksen er vertikal — derfor styrer align-items vertikal plassering. Vanlige verdier: flex-start, center, flex-end, stretch.

Mnemonisk: "justify = hovedakse (main axis), align = tverr-akse (cross axis)." Hvis du bytter til flex-direction: column, snur aksene — da styrer justify-content det som ser vertikalt ut.

Hvorfor de andre er feil: A styrer hovedaksen (her horisontal). C bestemmer om elementer kan bryte til ny linje. D er for hvor mye et element kan vokse.

Spørsmål 8 2 poeng Kap. 7 — alt-tekst

Hva bør alt-attributtet være for et rent dekorativt bilde (f.eks. en grafisk separator)?

  • A alt="dekorasjon"
  • B alt="bilde"
  • C Tom streng: alt=""
  • D Helt fjernet — bare <img src="..."> uten alt.
Vis fasit
Riktig svar: C

Tom alt="" forteller skjermlesere at bildet er rent dekorativt og kan hoppes over. Det er forskjellig fra å mangle attributtet helt — som signaliserer "alt-tekst mangler".

Spørsmål 10 3 poeng Kap. 9 — const-mutasjon

Hvilken av disse er gyldig JavaScript-kode (uten error)?

  • A const x = 5; x = 6;
  • B const a = [1,2]; a = [3,4];
  • C const a = [1,2]; a.push(3);
  • D const obj = {}; const obj = {x:1};
Vis fasit
Riktig svar: C

const hindrer reassignment av selve bindingen, ikke mutasjon av verdien den peker på. Et array som er deklarert med const kan fortsatt mutateres via metoder som push, pop, eller direkte indeksering.

Hvorfor de andre er feil: A reassigner en primitive — kaster TypeError: Assignment to constant variable. B reassigner array-bindingen til et nytt array — samme feil. D redeklarerer obj i samme scope — kaster SyntaxError.

Praktisk konsekvens: "const betyr ikke immutable." For ekte immutability må man fryse objektet med Object.freeze().

Spørsmål 11 4 poeng Kap. 9 — Array.reverse mutasjon

Hva skriver dette ut?

let a = [1, 2, 3];
let b = a.reverse();
console.log(a === b);
Vis fasit
Riktig svar: true

Mekanisme: reverse() muterer arrayet på stedet og returnerer en referanse til samme array. Derfor peker både a og b til det samme objektet i minnet — a === b er true.

Verdiene i begge er [3, 2, 1] — den opprinnelige rekkefølgen er borte fra a også.

Kontrasten verdt å huske: slice() returnerer en ny array (kopi), så a === a.slice() ville vært false. Det samme gjelder concat, map, filter.

Spørsmål 12 4 poeng Kap. 9 — String.indexOf med fromIndex

Hva skriver dette ut?

console.log("hallo verden".indexOf("l", 4));
Vis fasit
Riktig svar: -1

Mekanisme: indexOf(søkeord, fromIndex) starter søket fra indeks fromIndex. Strengen "hallo verden" har 'l' på indeks 2 og 3, men ingen 'l' etter indeks 4. Når søket ikke finner noe, returneres -1.

Indeksene er: h=0, a=1, l=2, l=3, o=4, (space)=5, v=6, e=7, r=8, d=9, e=10, n=11.

Vanlig feil: Glemme at fromIndex tas fra hvor søket starter, ikke "hopp over så mange treff". Tro at indexOf("l", 4) returnerer "den fjerde l-en".

Spørsmål 14 4 poeng Kap. 9 — parseInt med 0x-prefix
Ikke i pensum Forelesningene viste parseInt kun for desimaltall (Christmas-funksjonen). Auto-deteksjon av hex via 0x-prefiks er en detalj som ikke er gjennomgått — kommer ikke på eksamen. Behold som curiosa.

Hva skriver dette ut?

console.log(parseInt("0x1F"));
Vis fasit
Riktig svar: 31

Mekanisme: parseInt uten eksplisitt radix (basetall) auto-detekterer hex hvis strengen starter med 0x eller 0X. "1F" i basetall 16 er 1·16 + 15 = 31.

Anbefaling: Alltid send radix eksplisitt: parseInt("0x1F", 16) eller parseInt("42", 10). Uten radix har auto-deteksjonen historisk hatt edge cases (gamle nettlesere tolket "08" som oktalt).

Kontrast: Number("0x1F") gir også 31. Number("0x1F") + Number("0x1F") = 62.

Spørsmål 15 3 poeng Kap. 9 — Array.length-mutasjon

Hva skriver dette ut?

let a = [1, 2, 3, 4, 5];
a.length = 2;
console.log(a);
Vis fasit
Riktig svar: [1, 2]

Mekanisme: length er skrivbar i JS. Når du setter den til en lavere verdi, kutter JS arrayet (de overflødige elementene forsvinner). Når du setter den til en høyere verdi, fylles slutten med "tomme luker" (empty slots) som rapporterer undefined ved index-access.

Kontrast: a.length = 7 på [1,2] gir [1, 2, <5 empty items>] — en sparse array. Lengden er 7, men a[5] er undefined.

Verdien a[100] = 10 setter altså a.length til 101, ikke 2 — sett 01 har en variant av denne fellen.

Spørsmål 16 4 poeng Kap. 3 — External vs internal vs inline CSS

Forklar de tre måtene å legge til CSS på en HTML-side (external, internal, inline). Gi én konkret fordel og én konkret ulempe for hver.

Vis fasit

Rubrikk (4 poeng — ca. 1.3 poeng per metode pluss helhetsvurdering):

  • External: Egen .css-fil koblet via <link rel="stylesheet" href="...">. Fordel: kan deles av mange HTML-sider, caches av nettleseren. Ulempe: ekstra HTTP-request første gang.
  • Internal: CSS skrevet inne i <style>...</style> i <head>. Fordel: ingen ekstra request — alt i én fil. Ulempe: kan ikke gjenbrukes av andre sider, og blander struktur og styling.
  • Inline: CSS i style="..."-attributtet direkte på elementet. Fordel: overstyrer alt annet (specificity 1000), bra for engangsjusteringer. Ulempe: nesten ingen gjenbruk, vanskelig å vedlikeholde, bryter "separation of concerns".

Beste praksis: External er nesten alltid foretrukket for prosjekter med flere sider. Inline brukes typisk bare som override eller fra JS-genererte stiler.

Spørsmål 17 3 poeng Kap. 8 — POST vs GET (vinklet)

Du designer et innloggings-skjema som sender brukernavn og passord til serveren. Skal du bruke GET eller POST? Forklar valget med tre konkrete argumenter.

Vis fasit

Svar: POST. Rubrikk (3 poeng — 1 poeng per relevant argument):

  • Synlighet: GET legger parametere i URL-en — passord ville da være synlig i adresselinje, browser-historikk og server-logger. POST skjuler dem i request-body.
  • Caching og bookmarks: GET kan caches og bookmarkes — en lagret innloggings-URL ville være et katastrofalt sikkerhetsproblem. POST caches normalt ikke.
  • Sideeffekt: Innlogging endrer server-state (oppretter en bruker-sesjon). GET er ment for henteoperasjoner uten sideeffekter — bruk POST for noe som endrer noe.

Spørsmål 19 3 poeng Kap. 8 — placeholder vs label

En designer foreslår å fjerne alle <label>-elementer fra et skjema, og bare bruke placeholder="..." på input-feltene for å spare plass. Forklar minst tre grunner til at dette er en dårlig idé.

Vis fasit

Rubrikk (3 poeng — 1 poeng per godt argument):

  • Tilgjengelighet: Skjermlesere kobler <label for="..."> automatisk til input-feltet og annonserer det når feltet får fokus. Placeholder oppfattes ofte ikke som en label — brukere med screen reader får ikke vite hva feltet er for.
  • Placeholder forsvinner ved skriving: Så snart brukeren skriver noe, er placeholder borte — og hvis de glemmer hva feltet var, må de slette og se igjen.
  • Klikkeområde: Klikk på <label> fokuserer det tilhørende input-feltet. Det er nyttig spesielt på mobil (større touch-area). Placeholder gjør ikke noe når man klikker på den.
  • Lav kontrast: Placeholder-tekst rendres typisk som grå/lyse — kan være vanskelig å lese, spesielt for synshemmede.
  • Forveksles med utfylt verdi: En grå placeholder-tekst kan se ut som et allerede utfylt felt — brukere hopper over det og glemmer å fylle ut.

Spørsmål 20 5 poeng Kap. 10 — getElementById vs querySelector

Forklar forskjellen mellom document.getElementById("x"), document.getElementsByClassName("y"), document.querySelector(".y") og document.querySelectorAll(".y"). For hver: (a) hva returnerer den, (b) ett konkret eksempel, og (c) når den foretrekkes.

Vis fasit

Rubrikk (5 poeng — ~1.25 per metode):

  • getElementById("x"): Returnerer ett element (eller null hvis ingen finnes). Tar en id-streng uten #. Eksempel: getElementById("meny") finner <ul id="meny">. Foretrekkes når du vet at det er én unik id — raskest.
  • getElementsByClassName("y"): Returnerer en live HTMLCollection (oppdaterer seg automatisk når DOM-en endres). Eksempel: getElementsByClassName("aktiv") finner alle med class="aktiv". Iterables via index, men ikke via forEach direkte.
  • querySelector(".y"): Returnerer første element som matcher CSS-selektoren (eller null). Tar en full CSS-selektor med #, ., attributter, kombinatorer. Eksempel: querySelector("ul.meny > li.aktiv"). Foretrekkes når selektoren er kompleks.
  • querySelectorAll(".y"): Returnerer en static NodeList (snapshot, oppdateres ikke). Har forEach-metode direkte. Eksempel: querySelectorAll("input[type='text']").

Viktige skiller: Hash i selektoren (querySelector("#meny") men getElementById("meny")); live vs static collection; null vs tom collection.

Spørsmål 21 8 poeng Kap. 2+3 — Tabell-styling

HTML-en under inneholder en semantisk tabell. Skriv CSS som:

  1. (2 poeng) Gjør hele tabellen 100% bred og samler border-celler (border-collapse: collapse).
  2. (2 poeng) Gir alle celler (th og td) 1px grå ramme og 8px padding.
  3. (2 poeng) Gir <thead> bakgrunn #333 og hvit tekst.
  4. (2 poeng) Gir annenhver <tr> i <tbody> en lys bakgrunn (#f5f5f5) — bruk :nth-child(even).
Vis fasit

Referanseløsning:

table {
  width: 100%;
  border-collapse: collapse;
}

th, td {
  border: 1px solid #999;
  padding: 8px;
}

thead {
  background-color: #333;
  color: white;
}

tbody tr:nth-child(even) {
  background-color: #f5f5f5;
}

Kommentarer:

  • border-collapse: Uten denne får hver celle sin egen kant med mellomrom (default separate). collapse slår dem sammen til ett gitter — typisk det man vil ha.
  • Selektor for begge: th, td er en gruppe-selektor — én regel gjelder begge typer celler.
  • :nth-child(even): Treffer 2., 4., 6. <tr> innenfor parent. Alternativ: :nth-of-type(even) hvis det også finnes andre tags blandet inn.
  • Hvorfor "tbody tr": Uten den ville tr:nth-child(even) også treffe rader i thead — vi vil bare ha zebra-striper i datadelen.

Vanlige feil: Glemme border-collapse — kantene ser doble ut. Bruke cellpadding i HTML — det er deprecated. Bruke :nth-child(2n+1) som er odd, ikke even.

Spørsmål 22 8 poeng Kap. 10 — Sorter liste ved klikk

HTML-en under er gitt. Skriv JavaScript som:

  1. (3 poeng) Når brukeren klikker «Sorter A–Å», sorteres innholdet i <ul id="navne"> alfabetisk stigende.
  2. (3 poeng) Når brukeren klikker «Sorter Å–A», sorteres innholdet alfabetisk synkende.
  3. (2 poeng) Den eksisterende rekkefølgen erstattes (ikke duplisert) — så gjentatt klikking gir riktig resultat hver gang.
Vis fasit

Referanseløsning:

const liste = document.getElementById("navne");

function sortBy(retning) {
  // hent alle li, sorter, og legg tilbake i ny rekkefølge
  const items = Array.from(liste.querySelectorAll("li"));
  items.sort(function (a, b) {
    return retning === "opp"
      ? a.textContent.localeCompare(b.textContent)
      : b.textContent.localeCompare(a.textContent);
  });
  // erstatte innholdet: tøm først, så legg til på nytt
  liste.innerHTML = "";
  items.forEach(li => liste.appendChild(li));
}

document.getElementById("opp").addEventListener("click", () => sortBy("opp"));
document.getElementById("ned").addEventListener("click", () => sortBy("ned"));

Kommentarer:

  • Array.from: querySelectorAll returnerer NodeList som ikke har .sort(). Array.from(...) konverterer til ekte array som vi kan sortere.
  • localeCompare: Sortering med localeCompare håndterer norsk æøå korrekt — vanlig a < b-sammenligning ville plassert æ etter z, ikke etter å. (For en engelsk-eneste liste hadde a.textContent < b.textContent ? -1 : 1 også fungert.)
  • Hvorfor appendChild fungerer: Når du appendChild på et element som allerede er i DOM-en, blir det flyttet, ikke duplisert. Så vi kan innerHTML = "" for å rydde, og legge tilbake i ny rekkefølge.
  • Alternativ: Bygge en ny DOM, hente li som array via [...liste.children], eller bruke li.outerHTML. Mange løsninger er gyldige.

Vanlige feil: Sortere uten innerHTML = "" først → elementene legges til på toppen av de eksisterende, så listen vokser. Bruke .sort() direkte på NodeList (returnert av querySelectorAll) — det har ikke .sort()-metoden. Glemme localeCompare og få æøå feilplassert.

Spørsmål 23 7 poeng Kap. 9 — Factorial med rekursjon
Ikke i pensum Rekursjon (funksjon som kaller seg selv) er ikke gjennomgått i forelesningene — slidesene viser kun løkker. Eksamenstask vil bruke iterative løsninger. Bruk gjerne oppgaven til å øve på funksjons-syntaks; ignorer rekursjon-aspektet.

Skriv en JavaScript-funksjon fak(n) som beregner n! (n-fakultet) rekursivt. Reglene er:

  1. (3 poeng) Riktig rekursjon: fak(n) = n × fak(n-1), med base-case for n <= 1 som returnerer 1.
  2. (2 poeng) Koble funksjonen til knappen i HTML-en: når brukeren klikker, leses tallet fra input-feltet, og resultatet vises i <span id="ut">.
  3. (1 poeng) Håndter ugyldig input (tom, negativ, ikke-tall) med en passende feilmelding.
  4. (1 poeng) Bruk parseInt (eller Number) for å konvertere input-strengen til tall.
Vis fasit

Referanseløsning:

function fak(n) {
  if (n <= 1) return 1;          // base-case
  return n * fak(n - 1);          // rekursjon
}

document.getElementById("regn").addEventListener("click", function () {
  const raw = document.getElementById("tall").value;
  const n = parseInt(raw, 10);
  const ut = document.getElementById("ut");

  if (isNaN(n) || n < 0) {
    ut.textContent = "Ugyldig input — skriv et heltall ≥ 0";
    return;
  }
  ut.textContent = fak(n);
});

Kommentarer:

  • Base-case er kritisk: Uten if (n <= 1) return 1 ville rekursjonen aldri stoppet — JS ville hatt en stack overflow. Vi sjekker <= 1 (ikke === 0) for å håndtere både 0 og 1 (begge er 1 per definisjon).
  • parseInt med radix: Andre argument 10 sier "tolk som desimal". Uten det vil noen edge cases (typisk gamle nettlesere) tolke "08" som oktal.
  • isNaN-sjekken: parseInt("hei") returnerer NaN. NaN < 0 er false (NaN er ikke sammenlignbar), så sjekken må stå før intervall-sjekken.
  • Grenser: fak(20) er 2.4·1018 — fortsatt presist i JS. fak(170) overflower til Infinity. fak(10000) gir stack overflow uavhengig av talltype.

Vanlige feil: Glemme base-case (uendelig rekursjon). Returnere n * fak(n) i stedet for fak(n-1) (uendelig). Bruke parseFloat i stedet for parseInt (godtar desimal). Glemme parseInt — da blir "5" * fak("4") til "5" * 24 = 120 ved tilfeldighet, men bryter for andre tall.

Spørsmål 24 7 poeng Kap. 10 — Toggle panel-synlighet

HTML-en under har en knapp og en panel-div som er skjult i utgangspunktet. Skriv kode som:

  1. (2 poeng) Når brukeren klikker «Vis/skjul panel», skifter panelet mellom skjult og synlig (toggle).
  2. (2 poeng) Knappe-teksten endres til "Skjul panel" når panelet er synlig, og "Vis panel" når det er skjult.
  3. (2 poeng) Bruk en CSS-klasse .skjult (skriv den selv) for å styre synligheten via display: none — ikke direkte sett style i JS.
  4. (1 poeng) Bruk classList.toggle() eller classList.add/remove — ikke string-konkatener på className.
Vis fasit

Referanseløsning:

/* CSS */
.skjult { display: none; }

/* JS */
const bryter = document.getElementById("bryter");
const panel = document.getElementById("panel");

bryter.addEventListener("click", function () {
  panel.classList.toggle("skjult");
  // teksten følger ny state
  if (panel.classList.contains("skjult")) {
    bryter.textContent = "Vis panel";
  } else {
    bryter.textContent = "Skjul panel";
  }
});

Kommentarer:

  • classList.toggle: Returnerer true hvis klassen ble lagt til, false hvis den ble fjernet. Du kan også bruke det som verdi: const erSkjult = panel.classList.toggle("skjult"); bryter.textContent = erSkjult ? "Vis panel" : "Skjul panel";
  • Hvorfor ikke style.display: Da blandes CSS og JS. Hvis CSS-en endres (f.eks. til en animasjon med transform), må JS-en oppdateres også. Med klasse-strategien er JS kun ansvarlig for å sette/fjerne en flagg-klasse.
  • Alternativ klasse-navn: Det er vanlig å bruke .hidden, .is-hidden, eller .invisible. Navnet er sekundært — viktigere er at det er konsistent og beskriver tilstanden, ikke utseendet.

Vanlige feil: Bruke className += " skjult" — fjerner ikke klassen senere. Bruke style.display = "none" i JS — bryter separasjon av concerns og overstyrer CSS-en hardt. Glemme .skjult-regelen i CSS — knappen toggler klassen men ingenting visuelt skjer.

Spørsmål 25 10 poeng Kap. 8+9 — Komplett skjema-validering

Et adresseskjema er gitt under. Skriv JavaScript som validerer alle fire feltene når brukeren klikker «Send»:

  1. (2 poeng) Fornavn må bestå av kun bokstaver (a–z, æøå, A–Å) — ingen tall eller spesialtegn.
  2. (3 poeng) Postnummer må være nøyaktig 4 sifre mellom 0001 og 9999 (Norge-format).
  3. (2 poeng) Fødselsdato (åååå-mm-dd format) — feltet kan ikke være tomt.
  4. (2 poeng) Samtykke-checkboxen må være krysset av.
  5. (1 poeng) Hvis valideringen feiler, vis en samlet feilmelding i <p id="feil"> og hindre at skjemaet sendes. Hvis alt er OK, vis "Skjema validert!" i feltet.
Vis fasit

Referanseløsning:

document.getElementById("adr").addEventListener("submit", function (e) {
  e.preventDefault();
  const feilboks = document.getElementById("feil");
  const feil = [];

  const navn = document.getElementById("navn").value.trim();
  const post = document.getElementById("post").value.trim();
  const fdato = document.getElementById("fdato").value;
  const ok = document.getElementById("ok").checked;

  if (!/^[a-zA-ZæøåÆØÅ]+$/.test(navn)) {
    feil.push("Fornavn må bare ha bokstaver");
  }
  if (!/^\d{4}$/.test(post) || parseInt(post, 10) < 1) {
    feil.push("Postnummer må være 4 sifre (0001–9999)");
  }
  if (fdato === "") {
    feil.push("Velg fødselsdato");
  }
  if (!ok) {
    feil.push("Du må samtykke til vilkårene");
  }

  if (feil.length > 0) {
    feilboks.style.color = "crimson";
    feilboks.textContent = feil.join(". ") + ".";
  } else {
    feilboks.style.color = "green";
    feilboks.textContent = "Skjema validert!";
  }
});

Kommentarer:

  • Regex for navn: /^[a-zA-ZæøåÆØÅ]+$/ — ankrene ^ og $ sikrer at hele strengen er bokstaver. + krever minst ett tegn. Æ/Ø/Å må eksplisitt med — de er ikke en del av [a-z].
  • Regex for postnummer: /^\d{4}$/ — eksakt 4 sifre. parseInt(post, 10) < 1 ekskluderer "0000" som er ugyldig.
  • checkbox.checked: Returnerer boolean direkte — ingen string-konvertering nødvendig.
  • Samle feilmeldinger: Mye bedre brukeropplevelse å vise alle feil samtidig enn å stoppe ved første. Vi pusher til en array og join-er til slutt.
  • preventDefault umiddelbart: Vi kaller det først, og lar valideringa kjøre — selv om alt er OK, sender vi ikke skjemaet (siden vi ikke har en backend). I et reelt prosjekt ville man bare kalle preventDefault hvis det var feil.

Vanlige feil: Glemme æøå i navn-regex (et helt norsk navn som "Ådne" feiler). Glemme {4} i postnummer-regex så lengre tall passerer. Bruke checkbox.value i stedet for .checked (value er alltid "on" som er truthy uansett). Stoppe ved første feil — dårlig UX.

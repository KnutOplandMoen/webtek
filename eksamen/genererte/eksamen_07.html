Spørsmål 1 3 poeng Kap. 1 — URL-anatomi

Gitt URL-en https://www.example.com/news/article.html?id=42&lang=no#kommentarer. Hvilken del er spørringsstrengen (query string)?

  • A www.example.com
  • B /news/article.html
  • C ?id=42&lang=no
  • D #kommentarer
Vis fasit
Riktig svar: C

Query string-en starter med ? og består av nøkkel=verdi-par separert med &. Den brukes til å sende parametere til serveren — typisk fra GET-skjemaer eller for å filtrere/søke.

Hvorfor de andre er feil: A er domenenavnet. B er stien til ressursen (path). D er anker-fragmentet — det navigerer til et bestemt sted i dokumentet og sendes ikke til serveren.

Spørsmål 2 3 poeng Kap. 1 — IP-adresser

Hva er en IP-adresse i konteksten av Internett?

  • A Et menneskelig-lesbart domenenavn som ntnu.no.
  • B En unik numerisk identifikator for hver datamaskin på nettet — fire tall mellom 0 og 255.
  • C Et passord som klienten må sende ved hver HTTP-forespørsel.
  • D En type protokoll for å sende e-post mellom servere.
Vis fasit
Riktig svar: B

Hver maskin på Internett har en unik IP-adresse — fire tall i området 0–255, separert med punktum (f.eks. 129.241.160.102 for ntnu.no). Adressen brukes av rutere til å bestemme hvor pakkene skal sendes. DNS oversetter mellom domenenavn (A) og IP-adresse (B).

Hvorfor de andre er feil: A er domenenavnet — det DNS oversetter fra. C er konfabulasjon. D er SMTP (e-postprotokoll), ikke en adresse.

Spørsmål 3 3 poeng Kap. 2 — Heading-nivåer

Hva er den anbefalte praksisen for HTML-headinger (<h1>–<h6>) på en side?

  • A Bruk så mange <h1> som du vil for å gjøre teksten større.
  • B Bruk én <h1> som sidens hovedtittel, og bruk h2–h6 for å reflektere hierarkiet av seksjoner.
  • C Start alltid med <h6> og jobb deg oppover.
  • D Hopp over h2–h5 og bruk direkte h1 → h6 for å spare plass.
Vis fasit
Riktig svar: B

Heading-tags er semantiske — de beskriver hierarkiet i innholdet, ikke utseendet. Én <h1> per side er konvensjon (sidens hoved-emne). Under den følger h2 for hver hovedseksjon, h3 for underseksjoner under h2, osv.

Hvorfor det er viktig:

  • SEO: Søkemotorer prioriterer tekst i h1 og strukturen i h-tagene.
  • Tilgjengelighet: Skjermlesere lar brukeren navigere via heading-hierarki (hopp til neste h2). Bryter du hierarkiet ved å hoppe fra h1 til h4, blir det forvirrende.
  • Outlining: Verktøy som genererer innholdsfortegnelse baserer seg på h-strukturen.

Tips: Bruk CSS for å styre størrelse/farge — ikke velg h-tag basert på hvor stort du vil at det skal se ut.

Spørsmål 4 2 poeng Kap. 10 — innerHTML vs textContent

Du har en variabel navn = "<b>Ola</b>". Hva skjer hvis du setter elem.innerHTML = navn sammenlignet med elem.textContent = navn?

  • A Begge gjør samme — viser teksten "<b>Ola</b>" som vanlig.
  • B innerHTML tolker stringen som HTML — "Ola" vises i fet skrift. textContent viser den som ren tekst, inkludert < og > tegn.
  • C innerHTML viser som ren tekst; textContent tolker som HTML.
  • D Begge kaster en feil fordi <b> er en åpen tag uten lukkende.
Vis fasit
Riktig svar: B

innerHTML tolker stringen som HTML — så <b>-tagger blir faktiske bold-tagger og rendres som fet tekst. textContent behandler stringen som ren tekst — vinkelparentesene escapes og vises som "<b>Ola</b>" bokstavelig.

Sikkerhetshensyn: Hvis navn kommer fra brukerinput, kan en angriper sende <script>alert(1)</script>. Med innerHTML blir scriptet eksekvert (XSS-sårbarhet!). Med textContent vises bare den uskadelige teksten.

Tommelfingerregel: Bruk textContent hvis du bare skal sette tekst. Bruk innerHTML bare når du har full kontroll over innholdet (ingen brukerdata).

Spørsmål 5 3 poeng Kap. 10 — removeChild via parent

Du har const x = document.getElementById("mål");. Hva er den klassiske måten å fjerne x fra DOM-en?

  • A x.delete();
  • B document.removeElementById("mål");
  • C x.parentNode.removeChild(x);
  • D x.style.display = "none";
Vis fasit
Riktig svar: C

removeChild er en metode på forelderen, ikke på elementet selv. Klassisk syntaks: elementet.parentNode.removeChild(elementet) — vi går opp til forelderen og ber den fjerne barnet.

Moderne alternativ: x.remove() (uten parentNode-tur) er enklere og fungerer i alle moderne nettlesere. Bare ikke i IE11 eller eldre.

Hvorfor de andre er feil: A er konfabulering. B finnes ikke — det er getElementById, ingen removeElementById. D skjuler bare, fjerner ikke fra DOM-en — elementet finnes fortsatt og tar plass i hukommelsen.

Spørsmål 6 2 poeng Kap. 9 — typeof array

Hva returnerer typeof [1, 2, 3]?

  • A "array"
  • B "object"
  • C "number"
  • D "list"
Vis fasit
Riktig svar: B

Array er teknisk sett et spesielt objekt i JavaScript, og typeof returnerer derfor "object". Det samme gjelder {}, null, og dato-objekter.

For å sjekke om noe er en array: Bruk Array.isArray(x) — den er definitivt riktig. x instanceof Array fungerer også, men kan feile mellom iframes (forskjellige Array-konstruktører).

Spørsmål 7 3 poeng Kap. 2 — HTML5 sectioning vs div

Hva er hovedfordelen med <article> over <div> når du markerer en blog-post på en side?

  • A <article> har innebygd CSS-styling som <div> mangler.
  • B <article> gir semantisk informasjon til søkemotorer og skjermlesere om at innholdet er selvstendig og gjenbrukbart.
  • C <article> krever ikke lukkende tag.
  • D <article> rendres som inline-element i motsetning til <div> som er block.
Vis fasit
Riktig svar: B

HTML5 introduserte semantiske tags som <article>, <section>, <aside>, <nav>, <header>, <footer> for å beskrive innholdets rolle. Visuelt er de like som <div> by default, men semantisk forteller de søkemotorer, skjermlesere og andre verktøy hva delene betyr.

Hvorfor de andre er feil: A er feil — ingen innebygd CSS. C er feil — alle har lukkende tag. D er feil — begge er display: block default.

Når bruke article: Innhold som ville gi mening alene (kunne kopieres til et annet sted). Eksempler: blog-post, nyhet, forum-innlegg, produktkort. Ikke for generisk innpakning — der er <div> riktig valg.

Spørsmål 8 3 poeng Kap. 10 — createElement

Du vil lage et nytt <li> via JavaScript og legge det inn i en eksisterende <ul>. Hva er den riktige sekvensen?

  • A document.createTagName("li") deretter ul.append(li).
  • B document.createElement("li"); sett li.textContent = "..."; deretter ul.appendChild(li).
  • C document.newLi("...") direkte.
  • D ul.innerHTML = "<li>..." — det er den eneste måten.
Vis fasit
Riktig svar: B

Klassisk DOM-API for å lage og legge til et element:

  1. const li = document.createElement("li"); — lager en frittstående node (ikke i DOM ennå).
  2. li.textContent = "Tekst"; — setter innhold.
  3. ul.appendChild(li); — legger den inn på slutten av ul.

Moderne alternativ: ul.append(li) — fungerer også for tekst-strenger (ul.append("Tekst")) og flere argumenter. appendChild krever en node.

Hvorfor ikke innerHTML: D fungerer men fjerner alle eksisterende <li>-elementer (overskriver hele innholdet). Også har det XSS-risiko hvis tekstet kommer fra brukerinput.

Hvorfor de andre er feil: A bruker oppdiktet metode (createTagName finnes ikke). C bruker oppdiktet metode (newLi finnes ikke).

Spørsmål 9 2 poeng Kap. 1 — Klient-server-modell

I klient-server-modellen for web, hvem initierer alltid kommunikasjonen?

  • A Serveren — den "pusher" innhold til klienten.
  • B Klienten — den sender en request, serveren svarer med en response.
  • C Det avhenger av om det er HTTP/1.1 eller HTTP/2.
  • D Begge initierer samtidig via en handshake.
Vis fasit
Riktig svar: B

I HTTP er det alltid klienten (typisk en nettleser) som starter — sender request, venter på response. Serveren kan ikke "ringe opp" klienten av seg selv. Dette er en kjernekvalitet ved request/response-modellen.

Unntak: WebSockets bryter denne modellen — etter en initial handshake kan begge sider sende meldinger fritt. Men det er ikke "ekte HTTP" lenger.

Hvorfor de andre er feil: A er feil for HTTP. C er feil — selv HTTP/2 og HTTP/3 starter klient-initiert, selv om serveren kan "pushe" ekstra ressurser etterpå. D er feil — TCP-handshaken er klient-initiert (klienten sender SYN først).

Hvis du setter display: flex på et <span>-element, hvordan oppfører det seg?

  • A Ingenting endrer seg — display: flex krever et <div>.
  • B Spannet blir en flex-container, og dets barn legges ut i flex-layout. Spannet selv beholder en form for "block"-oppførsel.

Spørsmål 11 4 poeng Kap. 9 — String.substring negativ

Hva skriver dette ut?

console.log("WEBDESIGN".substring(3, -2));
Vis fasit
Riktig svar: WEB

Mekanisme — to triks i ett:

  • substring behandler negative argumenter som 0. Så -2 blir 0.
  • substring bytter også argumentene hvis start > end. Etter konverteringen er vi på substring(3, 0) → byttes til substring(0, 3).

Resultat: tegnene fra indeks 0 til (ikke inklusivt) 3 = "WEB".

Kontrast med slice: slice tolker negative argumenter som "tell fra slutten". "WEBDESIGN".slice(3, -2) ville gitt "DESI" (fra indeks 3 til 2 før slutt).

Klassisk eksamens-felle — krever at man husker disse to spesielle reglene for substring.

Spørsmål 12 4 poeng Kap. 9 — Array.push return

Hva skriver dette ut?

let a = [1, 2, 3];
let r = a.push(4, 5);
console.log(r);
Vis fasit
Riktig svar: 5

Mekanisme: push() returnerer den nye lengden til arrayet — ikke det som ble lagt til, og ikke arrayet selv. Etter push(4, 5) har vi [1, 2, 3, 4, 5] med lengde 5.

Vanlig feil: Tro at push returnerer det lagt-til elementet (det gjør pop). Eller tro at det returnerer arrayet selv (det gjør ikke det — kan brukes til kjede via lengden).

Return-verdier å huske:

  • push(x): ny lengde
  • unshift(x): ny lengde
  • pop(): fjernet element
  • shift(): fjernet element
  • splice(): array med fjernede elementer

Spørsmål 14 4 poeng Kap. 9 — Switch fall-through

Hva skriver dette ut?

let x = 2;
switch (x) {
  case 1:
    console.log("en");
  case 2:
    console.log("to");
  case 3:
    console.log("tre");
    break;
  case 4:
    console.log("fire");
}
Vis fasit
Riktig svar: to, tre

Mekanisme — switch fall-through: I JavaScript "renner" execution videre til neste case hvis det ikke er en eksplisitt break. Her starter vi på case 2 (treffer x=2) og skriver "to". Det er ingen break, så vi fortsetter inn i case 3 og skriver "tre". Der er det break, så vi stopper.

Vi traff aldri case 1 — case 1 ble hoppet over fordi x ≠ 1.

Hvorfor utviklere ofte bruker break overalt: Default fall-through-oppførselen er en av de vanligste kildene til subtile bugs. Det er gode use-cases (gruppere cases som skal gjøre samme), men de bør være eksplisitt kommentert.

Spørsmål 15 3 poeng Kap. 9 — null-coercion

Hva skriver dette ut?

console.log(null + 5);
Vis fasit
Riktig svar: 5

Mekanisme: Ved aritmetikk konverteres null til 0. Så null + 5 = 0 + 5 = 5.

Inkonsistens med undefined: undefined + 5 derimot er NaN, fordi undefined ikke har en numerisk konvertering. Disse to "tomme" verdiene oppfører seg forskjellig — en av JavaScripts klassiske quirks.

Andre eksempler:

  • null + null → 0
  • null + "5" → "null5" (string-konkat, ikke aritmetikk)
  • null == undefined → true (særskilt regel)
  • null === undefined → false (strict, ulike typer)

Spørsmål 16 5 poeng Kap. 1 — Full TCP/IP-modell

Forklar de fire lagene i TCP/IP-modellen. For hvert lag: hva er rollen, og gi to konkrete protokoll-eksempler. Dette er et tier-1-spørsmål som dukker opp i nesten alle gamle eksamener.

Vis fasit

Rubrikk (5 poeng — ca. 1.2 poeng per lag, pluss helhetsvurdering):

  • 1. Applikasjonslag (Application layer) — øverst: Hvor selve nettverks-applikasjonene kommuniserer. Protokoller som tolkes av nettlesere, mail-klienter, etc. Eksempler: HTTP, FTP, DNS, telnet, VoIP, SMTP/IMAP/POP3.
  • 2. Transportlag (Transport layer): Bryter data opp i pakker, sørger for at de når frem og settes sammen igjen i riktig rekkefølge. Tilbyr porter for å skille mellom applikasjoner. Eksempler: TCP (pålitelig, ordnet, krever forbindelse), UDP (rask, upålitelig, ingen forbindelse — brukes for streaming, DNS-oppslag).
  • 3. Nettverkslag (Network/Internet layer): Rutet pakker mellom maskiner basert på IP-adresser. "Hvordan kommer denne pakken fra min maskin til serveren i USA?" Eksempler: IP (versjon 4 og 6), ICMP (brukt av ping).
  • 4. Fysisk/Link-lag (Physical/Link layer) — nederst: Den faktiske maskinvaren — hvordan bits sendes som elektriske signaler, radiosignaler, lysblink i fiber. Eksempler: Ethernet, WiFi (802.11), Bluetooth, fiber.

Husk-trick: Akronymer som "ATNF" (Applikasjon, Transport, Nettverk, Fysisk) eller bruk hverdagseksempler: en HTTP-request reiser ned gjennom alle fire lagene på din maskin, over kabel/luft, opp gjennom de samme fire lagene på serveren. Hvert lag legger til sin "konvolutt" på vei ned, og fjerner den på vei opp.

Bonus: Forskjellen mellom TCP og UDP er ofte verdt å trekke fram — TCP er som en telefonsamtale (forbindelse, ordnet), UDP er som postkort (du sender og håper).

Spørsmål 18 3 poeng Kap. 10 — innerHTML vs textContent vs innerText

Forklar forskjellen mellom element.innerHTML, element.textContent og element.innerText. Gi ett scenario hvor du foretrekker hver av dem.

Vis fasit

Rubrikk (3 poeng):

  • innerHTML: Returnerer/setter HTML-strukturen som string, inkludert tagger. Set-versjonen parser stringen som HTML. Foretrekkes når: du vil sette HTML-fragmenter du har full kontroll over (f.eks. mal som du bygger som template literal).
  • textContent: Returnerer/setter ren tekst — ingen HTML-parsing, ingen skjult tekst-filtering. Returnerer også tekst i hidden elementer (display:none). Foretrekkes når: du setter brukerinput eller andre untrusted strings; eller når du vil ha all tekst inkludert skjult.
  • innerText: Som textContent, men respekterer CSS — returnerer kun synlig tekst. Hvis et element er display: none, inkluderes ikke teksten. Mer tregt fordi det krever layout-beregning. Foretrekkes når: du vil ha akkurat det brukeren ser, f.eks. "kopier til utklippstavle"-funksjonalitet.

Tommelfingerregel:

  • Bytt tekst → textContent (raskeste, tryggeste)
  • Bygg HTML-struktur fra mal → innerHTML (sjekk for XSS)
  • Få det brukeren ser → innerText (sjelden brukt)

Spørsmål 19 3 poeng Kap. 2 — DOM-tre

Forklar hva DOM (Document Object Model) er, og hvordan en HTML-side blir til et tre. Bruk et minimalt eksempel:

<p>Hei <b>verden</b>!</p>

Tegn/beskriv DOM-treet for dette utdraget, og forklar hva en JS-kode som document.querySelector("b").textContent ville returnere.

Vis fasit

Rubrikk (3 poeng):

    p
├── tekstnode "Hei "
├── b
│   └── tekstnode "verden"
└── tekstnode "!"

HTML-en under har en tabell med en header-rad og to inputfelt. Skriv kode som:

  1. (3 poeng) Når brukeren klikker «Legg til», lager du en ny rad i <tbody> med to celler — den ene fra navn-input, den andre fra alder-input.
  2. (2 poeng) Tøm inputfeltene etter at raden er lagt til.
  3. (2 poeng) Hvis enten navn eller alder er tomt, ikke legg til raden — vis en advarsel i alert.
  4. (1 poeng) Bruk createElement + appendChild, ikke innerHTML +=.
Vis fasit

Referanseløsning:

const navnInput = document.getElementById("navn");
const alderInput = document.getElementById("alder");
const tbody = document.getElementById("tbody");

document.getElementById("legg").addEventListener("click", function () {
  const navn = navnInput.value.trim();
  const alder = alderInput.value.trim();

  if (navn === "" || alder === "") {
    alert("Begge felter må fylles ut");
    return;
  }

  const tr = document.createElement("tr");
  const tdNavn = document.createElement("td");
  const tdAlder = document.createElement("td");
  tdNavn.textContent = navn;
  tdAlder.textContent = alder;
  tr.appendChild(tdNavn);
  tr.appendChild(tdAlder);
  tbody.appendChild(tr);

  // tøm input-feltene
  navnInput.value = "";
  alderInput.value = "";
});

Kommentarer:

  • trim() før sjekk: " " skal ikke regnes som gyldig input. trim() fjerner kantverdig whitespace.
  • createElement-mønsteret: Lag noden, sett innholdet, append til forelderen. For mer komplekse strukturer er det vanlig å lage en helper-funksjon.
  • textContent vs innerHTML: Vi bruker textContent for å unngå at brukerinput som "<script>..." får tolket som HTML.
  • Hvorfor ikke innerHTML +=: Hver gang du gjør tbody.innerHTML += "...", parses hele innholdet på nytt. Hvis det er event-listeners på eksisterende rader, går de tapt. Bedre å appende noder eksplisitt.

Vanlige feil: Glemme trim(), så " " teller som gyldig. Bruke tbody.innerHTML += "<tr>..." — fungerer, men er fragilt. Glemme å tømme inputfeltene.

Spørsmål 22 8 poeng Kap. 9+10 — Fibonacci-funksjon

Skriv en JavaScript-funksjon fib(n) som returnerer det n-te Fibonacci-tallet (0-indeksert). Reglene: fib(0) = 0, fib(1) = 1, fib(n) = fib(n-1) + fib(n-2). Knytt funksjonen til knappen i HTML-en.

  1. (4 poeng) Skriv en iterativ implementasjon (med løkke). Riktig: fib(6) = 8, fib(10) = 55.
  2. (2 poeng) Knytt funksjonen til knappen — les n fra input, vis resultatet i <span id="ut">.
  3. (1 poeng) Håndter ugyldig input (negativ, ikke-tall) med feilmelding.
  4. (1 poeng) Bruk parseInt for å konvertere input.
Vis fasit

Referanseløsning (iterativ):

function fib(n) {
  if (n === 0) return 0;
  if (n === 1) return 1;
  let a = 0;
  let b = 1;
  for (let i = 2; i <= n; i++) {
    const ny = a + b;
    a = b;
    b = ny;
  }
  return b;
}

document.getElementById("kjor").addEventListener("click", function () {
  const n = parseInt(document.getElementById("n").value, 10);
  const ut = document.getElementById("ut");
  if (isNaN(n) || n < 0) {
    ut.textContent = "Ugyldig — skriv et heltall ≥ 0";
    return;
  }
  ut.textContent = fib(n);
});

Kommentarer:

  • Hvorfor iterativ over rekursiv: En naiv rekursiv fib (uten memoisering) har eksponensiell kjøretid — fib(40) tar flere sekunder. Iterativ er O(n) lineær tid. For Fibonacci anbefales iterativ.
  • Rolling sum med to variabler: Vi trenger bare de to siste tallene, ikke hele historikken. a og b holder de to forrige; ny er det neste.
  • Base-cases: Vi sjekker n === 0 og n === 1 opp front. Uten dem ville løkkas for i=2; i<=n aldri kjørt for n < 2, men b ville være tilstandig 1 — feil svar for n=0.
  • Alternativ rekursiv (men sakte): 
    function fib(n) {
  if (n < 2) return n;
  return fib(n-1) + fib(n-2);
}
    Fungerer korrekt, men dør på store n.

Vanlige feil: Glemme base-case for n=0 (returnerer 1 i stedet for 0). Bytte rekkefølge av a = b og b = ny — da blir a det nye, ikke det forrige. Bruke rekursjon uten å innse at det blir tregt for n > 35.

HTML-en under har et passord-felt og en bekreft-passord-felt. Skriv kode som validerer fortløpende (på hvert tastetrykk):

  1. (3 poeng) Vis "For kort" hvis passordet er under 8 tegn, "OK" ellers, i <span id="p1Status">.
  2. (3 poeng) Vis "Stemmer" hvis bekreftelses-feltet er likt passordet, "Stemmer ikke" ellers, i <span id="p2Status">.
  3. (2 poeng) Endre tekstfargen: "OK"/"Stemmer" i grønt, "For kort"/"Stemmer ikke" i rødt.
  4. (2 poeng) Knappen «Send» skal være disabled hvis passordet er ugyldig eller bekreftelsen ikke stemmer.
Vis fasit

Referanseløsning:

const p1 = document.getElementById("p1");
const p2 = document.getElementById("p2");
const p1Status = document.getElementById("p1Status");
const p2Status = document.getElementById("p2Status");
const sendBtn = document.getElementById("send");

function valider() {
  // sjekk lengde
  const lengdeOk = p1.value.length >= 8;
  if (lengdeOk) {
    p1Status.textContent = "OK";
    p1Status.style.color = "green";
  } else {
    p1Status.textContent = "For kort";
    p1Status.style.color = "crimson";
  }

  // sjekk match
  const matchOk = p1.value === p2.value && p2.value !== "";
  if (matchOk) {
    p2Status.textContent = "Stemmer";
    p2Status.style.color = "green";
  } else {
    p2Status.textContent = "Stemmer ikke";
    p2Status.style.color = "crimson";
  }

  // styr knappen
  sendBtn.disabled = !(lengdeOk && matchOk);
}

p1.addEventListener("input", valider);
p2.addEventListener("input", valider);

Kommentarer:

  • input-event: Trigges på hvert tastetrykk, paste, autocomplete. Bedre enn keyup her fordi det fanger også paste-operasjoner.
  • Felles valider-funksjon: Begge inputs trigger samme funksjon. Vi sjekker både felter hver gang, slik at endring i p1 også oppdaterer match-statusen i p2.
  • Sjekk på p2.value !== "": Uten denne ville matchen være "true" når begge felter er tomme — misvisende. Vi krever at p2 har noe innhold før vi sier "Stemmer".
  • disabled-property: En boolean. !(lengdeOk && matchOk) betyr "ikke begge OK". Brukeren kan ikke klikke knappen før alt stemmer.

Vanlige feil: Bruke change-event — trigges først ved blur, ikke i sanntid. Glemme tom-string-sjekken så "" og "" matcher. Sette style.color bare ved feil, så ved retting fortsetter den å være rød.

Spørsmål 25 7 poeng Kap. 10 — Klassisk kalkulator (mini)

HTML-en under har to inputfelt og fire knapper for de fire grunnregningene. Skriv JavaScript som:

  1. (2 poeng) Hver knapp leser de to tallene fra inputfeltene og gjør riktig operasjon (+, -, *, /).
  2. (2 poeng) Resultatet vises i <span id="res">.
  3. (2 poeng) Hvis brukeren prøver å dele på 0, vis "Kan ikke dele på null".
  4. (1 poeng) Bruk parseFloat for å håndtere desimaltall.
Vis fasit

Referanseløsning:

const aFelt = document.getElementById("a");
const bFelt = document.getElementById("b");
const res = document.getElementById("res");

function regn(op) {
  const a = parseFloat(aFelt.value);
  const b = parseFloat(bFelt.value);
  if (isNaN(a) || isNaN(b)) {
    res.textContent = "Ugyldig input";
    return;
  }
  let svar;
  switch (op) {
    case "+": svar = a + b; break;
    case "-": svar = a - b; break;
    case "*": svar = a * b; break;
    case "/":
      if (b === 0) { res.textContent = "Kan ikke dele på null"; return; }
      svar = a / b;
      break;
  }
  res.textContent = svar;
}

document.getElementById("pluss").addEventListener("click", () => regn("+"));
document.getElementById("minus").addEventListener("click", () => regn("-"));
document.getElementById("gange").addEventListener("click", () => regn("*"));
document.getElementById("del").addEventListener("click", () => regn("/"));

Kommentarer:

  • Felles regn-funksjon: Vi unngår å duplikere parseFloat-logikken fire ganger. Hver knapp er bare en tynn wrapper som spesifiserer hvilken operasjon.
  • parseFloat over parseInt: Brukeren kan skrive 2.5 / 4.7 etc. parseInt ville kuttet desimalene.
  • Division by zero: JS returnerer faktisk Infinity for a/0 (eller -Infinity), ikke en feil. Men semantisk er det ugyldig — så vi sjekker eksplisitt.
  • Arrow functions for handlere: () => regn("+") er ekvivalent med function() { regn("+"); } men kortere. Vi trenger ikke this-kontroll her.

Vanlige feil: Glemme parseFloat — "6" + "3" blir "63". Bruke samme handler-funksjon på alle knappene uten å vite hvilken som ble klikket. Glemme division-by-zero-sjekken.

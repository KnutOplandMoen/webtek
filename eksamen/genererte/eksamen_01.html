Spørsmål 1 3 poeng Kap. 1 — TCP/IP

Hvilken av disse protokollene hører hjemme i nettverkslaget (Network layer) i TCP/IP-modellen?

  • A TCP
  • B HTTP
  • C IP
  • D Ethernet
Vis fasit
Riktig svar: C

IP (Internet Protocol) er selve nettverkslaget — det står for å rute pakker fra avsender til mottaker basert på IP-adresser.

Hvorfor de andre er feil: TCP er på transportlaget (over IP). HTTP er på applikasjonslaget (helt øverst). Ethernet er på fysisk/link-laget (helt nederst).

Spørsmål 2 2 poeng Kap. 1 — HTTP-statuskoder

Hva er forskjellen mellom HTTP-statuskode 301 og 302?

  • A 301 er permanent redirect; 302 er midlertidig redirect.
  • B 301 betyr "Forbidden"; 302 betyr "Not Found".
  • C 301 er midlertidig redirect; 302 er permanent redirect.
  • D 301 brukes ved POST; 302 brukes ved GET.
Vis fasit
Riktig svar: A

301 ("Moved Permanently") forteller nettleseren at ressursen er flyttet for godt, og at bookmarks/lenker bør oppdateres. 302 ("Found"/midlertidig) sier at ressursen er flyttet akkurat nå, men kan komme tilbake — bookmarks skal ikke oppdateres.

Hvorfor de andre er feil: B blander 301/302 med 403/404. C har snudd betydningene. D er en konfabulering — redirect-typen er uavhengig av request-metoden.

Hvilken påstand om id- og class-attributtene i HTML er korrekt?

  • A Både id og class kan brukes flere ganger i samme dokument.
  • B id skal være unik per dokument; class kan brukes på mange elementer.
  • C class skal være unik per dokument; id kan brukes på mange elementer.
  • D Begge er unike per element, men kan deles mellom elementer hvis CSS tillater det.
Vis fasit
Riktig svar: B

HTML-spesifikasjonen sier id må være unik i hele dokumentet — det er en identifikator for ett spesifikt element. class er en gruppeetikett som kan settes på så mange elementer du vil.

Spørsmål 4 3 poeng Kap. 3 — Box-model / shorthand

Hvilken regel beskriver hva padding: 10px 20px; gjør i CSS?

  • A 10px på alle fire sider; 20px legges til som ekstra margin.
  • B 10px top, 20px right, 10px bottom, 20px left (top og bottom = første verdi).
  • C 10px left, 20px right, 0 top og bottom.
  • D 10px på alle fire sider, og 20px ignoreres som syntaksfeil.
Vis fasit
Riktig svar: B

Ved to verdier gjelder regelen vertikal–horisontal: første verdi er top og bottom (10px), andre verdi er right og left (20px). Mnemonic: "TBLR-klokken med to verdier" — vertikalt først, horisontalt etterpå.

Hvorfor de andre er feil: A blander padding med margin. C er feil rekkefølge (left ≠ 10). D er feil — to verdier er gyldig shorthand-syntaks.

Hva skriver følgende kode ut?

Vis fasit

Spørsmål 12 3 poeng Kap. 9 — Array.sort()

Hva skriver koden under ut?

let nums = [10, 2, 5];
nums.sort();
console.log(nums[0]);
Vis fasit
Riktig svar: 10

Mekanisme — den klassiske sort-fellen: Uten compare-funksjon konverterer sort() elementene til strenger og sammenligner leksikografisk. Tegnet "1" kommer før "2" og "5" alfabetisk, så "10" sorteres før "2". Resultat: [10, 2, 5], og nums[0] er 10.

For å få numerisk sortering må man bruke nums.sort((a, b) => a - b), som ville gitt [2, 5, 10] og resultat 2.

Spørsmål 15 3 poeng Kap. 9 — Type-coercion

Hva skriver dette ut?

console.log("3" * "4");
Vis fasit
Riktig svar: 12

Mekanisme: Operatoren * har ikke noen string-overload (i motsetning til +, som konkatenerer strenger). Den tvinger derfor begge operandene til tall: "3" → 3, "4" → 4, og regner 3 * 4 = 12.

Kontrasten verdt å huske: "3" + "4" blir "34" (konkatenert streng), mens "3" - "4" blir -1 og "3" / "4" blir 0.75. Bare + har den tvetydige oppførselen.

Spørsmål 16 3 poeng Kap. 9 — Array.splice mutasjon

Hva skriver dette ut?

let arr = [1, 2, 3, 4, 5];
arr.splice(1, 2);
console.log(arr.length);
Vis fasit
Riktig svar: 3

Mekanisme: splice(start, deleteCount) muterer arrayet — den fjerner deleteCount elementer fra og med start. Her: fra indeks 1, fjern 2 elementer (verdiene 2 og 3). Resultat: arr = [1, 4, 5], og arr.length er 3.

Viktig kontrast — splice vs slice: slice muterer ikke; den returnerer en kopi. arr.slice(1, 2) ville returnert [2] uten å endre arr.

Spørsmål 17 3 poeng Kap. 8 — POST vs GET

Forklar minst tre konkrete forskjeller mellom GET og POST når du sender skjema-data.

Vis fasit

Sensor leter etter minst tre av disse punktene (1 poeng per godt punkt, maks 3):

  • Plassering: GET legger parameterne i URL-en (etter ?), POST legger dem i request-body — usynlige i URL.
  • Idempotens og caching: GET er idempotent og kan caches/bookmarks/legges i browser-historikk. POST er ikke idempotent og caches normalt ikke.
  • Størrelse: GET er praktisk begrenset til ~2 KB (URL-lengde-grense i nettlesere/servere). POST har ingen praktisk grense.
  • Synlighet: GET-data vises i URL, history, server-logger — uegnet for sensitive data. POST-data er skjult i body (men ikke kryptert med mindre HTTPS brukes).
  • Bruksområder: GET for henting/lesing (søk, filtre); POST for å endre serverens tilstand (registrering, sending av skjema).

Vis fasit

Spørsmål 19 3 poeng Kap. 10 — Event-handlere

Hva er tre konkrete fordeler ved å bruke element.addEventListener("click", fn) framfor inline onclick="fn()" i HTML?

Vis fasit

Sensor leter etter minst tre av disse (1 poeng hver, maks 3):

  • Flere handlere på samme event: addEventListener tillater å registrere mange callbacks for samme event; inline onclick overskriver foregående handler.
  • Separasjon av structure og behavior: All JS holdes i .js-filer; HTML-en forblir ren markup. Lettere å vedlikeholde.
  • Capturing-fasen tilgjengelig: Tredje argument true aktiverer capturing — ikke mulig via inline-syntaksen.
  • Lett å fjerne: removeEventListener tar samme funksjons-referanse og rydder opp. Inline må manuelt nulles ut.
  • Tilgang til Event-objektet: Funksjonen får event-parameteren automatisk; man kan kalle e.preventDefault() og e.stopPropagation() rent.

Spørsmål 24 10 poeng Kap. 8+9 — Form-validering

HTML-skjemaet under er gitt. Skriv JavaScript som validerer feltene når brukeren klikker «Send»:

  1. (3 poeng) Email må matche en enkel regex som krever et @-tegn med tekst på begge sider og minst ett punktum etter @. Eksempel: kari@ntnu.no godkjennes, kari@no ikke.
  2. (3 poeng) Telefonnummer må være nøyaktig 8 sifre, uten mellomrom eller andre tegn.
  3. (2 poeng) Alder må være et heltall mellom 16 og 100 (inklusive).
  4. (2 poeng) Vis feilmeldinger i alert() og hindre at skjemaet sendes hvis noe er ugyldig.
Vis fasit

Referanseløsning:

const form = document.getElementById("reg");
form.addEventListener("submit", function (e) {
  const email = document.getElementById("email").value;
  const tel = document.getElementById("tel").value;
  const alder = parseInt(document.getElementById("alder").value, 10);

  const emailRegex = /^[^@\s]+@[^@\s]+\.[^@\s]+$/;
  if (!emailRegex.test(email)) {
    alert("Ugyldig email-adresse");
    e.preventDefault();
    return;
  }

  if (!/^\d{8}$/.test(tel)) {
    alert("Telefon må være nøyaktig 8 sifre");
    e.preventDefault();
    return;
  }

  if (isNaN(alder) || alder < 16 || alder > 100) {
    alert("Alder må være et heltall mellom 16 og 100");
    e.preventDefault();
    return;
  }
});

Kommentarer:

  • Email-regex: /^[^@\s]+@[^@\s]+\.[^@\s]+$/ krever at noe står før @, noe står mellom @ og ., og noe etter .. [^@\s]+ betyr "ett eller flere tegn som ikke er @ eller whitespace". Det finnes mer presise email-regex-er, men denne er rimelig for en eksamen-besvarelse.
  • Telefon-regex: /^\d{8}$/ krever nøyaktig 8 sifre fra start til slutt. Ankrene ^ og $ er viktige — uten dem ville "12345678ab" også godkjennes.
  • Alder: parseInt(...) returnerer NaN hvis input ikke kan tolkes som tall. isNaN() fanger det. Sjekk også begge endene av intervallet.
  • preventDefault(): Uten den ville skjemaet faktisk submittes uansett. return false hadde også fungert i en inline-handler, men ikke her med addEventListener.

Vanlige feil: Bruke type="email" i HTML (nettleseren validerer selv, men oppgaven krever JS-validering). Glemme preventDefault() — skjemaet sendes likevel. Bruke type="number" for alder (gjør validering enklere, men oppgaven sier text). Email-regex som er for streng eller for slapp.

Skriv en JavaScript-funksjon tilMidnatt() som regner ut antall minutter som er igjen til neste midnatt (00:00 neste dag) og viser resultatet i en alert(). Funksjonen skal kjøres når man klikker på knappen i HTML-en under. Bonusinformasjon i alert-en: er antallet minutter et oddetall eller partall?

  1. (3 poeng) Riktig beregning av minutter til midnatt — bruk new Date() og en ny Date for neste midnatt.
  2. (1 poeng) Korrekt rounding (avrund nedover med Math.floor).
  3. (1 poeng) Sjekk oddetall/partall med %.
  4. (1 poeng) Alert med en lesbar melding som inneholder begge deler.
Vis fasit

Referanseløsning:

function tilMidnatt() {
  const naa = new Date();
  const midnatt = new Date(
    naa.getFullYear(),
    naa.getMonth(),
    naa.getDate() + 1,  // neste dag
    0, 0, 0             // 00:00:00
  );
  const ms = midnatt - naa;
  const minutter = Math.floor(ms / (1000 * 60));
  const paritet = (minutter % 2 === 0) ? "partall" : "oddetall";
  alert(minutter + " minutter til midnatt — " + paritet);
}

document.getElementById("vis").addEventListener("click", tilMidnatt);

Kommentarer:

  • Date-konstruktøren: new Date(år, måned, dag, time, minutt, sekund) — månedsindeks er 0–11, dag er 1–31. Ved å sette getDate() + 1 og time/minutt/sekund til 0 får du midnatt i begynnelsen av neste dag.
  • Subtraksjon på Date: midnatt - naa tvinger begge til millisekunder (siden epoch). Resultatet er millisekunder til midnatt.
  • Konvertering: Del på 1000 * 60 for å gå fra ms til minutter. Math.floor avrunder nedover.
  • Paritet: x % 2 gir 0 for partall og 1 for oddetall.

Vanlige feil: Bruke getDate() uten +1 (gir negativt tall hvis klokka er etter midnatt). Bruke Math.round som kan runde opp og gi 1440 minutter (=24 timer, ikke riktig). Bruke Math.ceil som gir samme problem hvis ms er nær et helt minutt.

HTML-en under er gitt. Skriv JavaScript som gjør følgende:

  1. (3 poeng) Når brukeren klikker «Legg til», skal teksten fra input-feltet legges til som en ny <li> nederst i <ul id="liste">. Input-feltet skal tømmes etterpå.
  2. (2 poeng) Hvis input-feltet er tomt (eller bare inneholder whitespace), skal ingenting legges til.
  3. (2 poeng) Når brukeren klikker på en eksisterende <li> (også de som er lagt til dynamisk), skal den fjernes fra listen.
Vis fasit

Referanseløsning (idiomatisk variant med event delegation):

const input = document.getElementById("nytt");
const legg = document.getElementById("legg-til");
const liste = document.getElementById("liste");

legg.addEventListener("click", function () {
  const tekst = input.value.trim();
  if (tekst === "") return;          // hopp over tom input
  const li = document.createElement("li");
  li.textContent = tekst;
  liste.appendChild(li);
  input.value = "";                  // tøm input
});

liste.addEventListener("click", function (e) {
  if (e.target.tagName === "LI") {
    e.target.remove();
  }
});

Kommentarer:

  • Event delegation: Vi lytter på <ul> i stedet for hver <li>. Det er hele poenget — slik fungerer det også for dynamisk lagt til li-elementer uten at vi må feste handlere på dem individuelt. Bubbling lar klikket boble opp til <ul>, og vi sjekker e.target for å se hva som ble klikket.
  • Trim: "".trim() og " ".trim() blir begge tomme strenger, så sjekken dekker både helt tom og bare whitespace.
  • textContent vs innerHTML: Bruk textContent hvis du bare skal sette tekst — det er tryggere mot ufrivillig HTML-injection.

Vanlige feil: Feste klikk-handler bare på de eksisterende <li>-ene — nye lagt til får ikke handleren. Bruke innerHTML += "<li>..." — dette gjenoppretter alle li-elementer og fjerner derved event-handlere. Glemme trim() så bare-whitespace passerer.

Lag en JavaScript-funksjon som styrer hover-oppførselen til boksen i HTML-en under — uten å bruke CSS :hover. Boksen skal:

  1. (2 poeng) Endre bakgrunnsfarge til lyseblå (#cfe3f5 e.l.) når musepekeren går over den (event mouseover eller mouseenter).
  2. (2 poeng) Endre tekstinnholdet til "Du holder musa over!" mens musa er over.
  3. (2 poeng) Resette begge til original bakgrunn (#eee) og original tekst når musa forlater boksen.
  4. (1 poeng) Bruke addEventListener (ikke inline onmouseover).
Vis fasit

Referanseløsning:

const boks = document.getElementById("boks");
const originalTekst = boks.textContent;

boks.addEventListener("mouseover", function () {
  boks.style.backgroundColor = "#cfe3f5";
  boks.textContent = "Du holder musa over!";
});

boks.addEventListener("mouseout", function () {
  boks.style.backgroundColor = "#eee";
  boks.textContent = originalTekst;
});

Kommentarer:

  • Hvorfor lagre original tekst: Hvis vi hardkoder "Hold musa over meg" i JS-en, må vi endre to steder hvis HTML-en endres. Lagre den én gang ved siden lastes og bruk den variabelen.
  • mouseover vs mouseenter: Begge fungerer her. Forskjellen: mouseover bobler, mouseenter ikke — så hvis boksen inneholder barn-elementer kan mouseover trigge flere ganger når musa går over de indre elementene. mouseenter trigger bare én gang ved selve grensen.
  • style.backgroundColor: Camel-case fordi background-color i CSS blir backgroundColor i DOM-API-en (bindestrek blir ikke gyldig JS-navn).

Vanlige feil: Hardkode original tekst. Bruke style.background-color (bindestrek, ikke gyldig JS). Bruke inline onmouseover="..." i HTML — gir 0 av 1 poeng på deloppgave 4. Glemme å resette ved mouseout slik at boksen er låst i over-tilstand.

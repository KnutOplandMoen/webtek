Hva er den primære funksjonelle forskjellen mellom <input type="text"> og <input type="email">?

  • A type="email" krypterer verdien automatisk ved submit.
  • B type="email" aktiverer e-post-spesifikt mobiltastatur og innebygd format-validering ved submit.

Spørsmål 5 3 poeng Kap. 9 — Hoisting med var
Ikke i pensum «Hoisting» som konsept (deklarasjoner heves til toppen av scope) er ikke gjennomgått i forelesningene. Slidesene nevner kun at var tillater re-deklarasjon — derfor bruk let. Hopp over hvis du øver til eksamen.

Hva skriver dette ut?

console.log(x);
var x = 5;
console.log(x);
  • A 5 og deretter 5
  • B ReferenceError ved første linje
  • C undefined og deretter 5
  • D null og deretter 5
Vis fasit
Riktig svar: C

JavaScript "hoister" var-deklarasjoner til toppen av scope, men ikke tildelingen av verdi. Etter hoisting ser koden slik ut:

var x;            // hoisted deklarasjon
console.log(x);   // x finnes, men er undefined
x = 5;            // selve tildelingen
console.log(x);   // 5

Hvorfor de andre er feil: A glemmer at tildelingen ikke hoister. B ville vært riktig med let eller const — de er i "temporal dead zone" før deklarasjonen. D er feil — JS gir undefined, ikke null.

Praktisk konsekvens: Hoisting er en av grunnene til at let/const er foretrukket — de gir ReferenceError og avslører feilen tidlig.

Spørsmål 6 3 poeng Kap. 9 — NaN-sammenligning

Hva returnerer NaN === NaN i JavaScript?

  • A true — fordi de begge representerer samme "not a number"-verdi.
  • B false — NaN er per definisjon ikke lik noe, heller ikke seg selv.
  • C undefined — sammenligningen er udefinert.
  • D Kaster TypeError.
Vis fasit
Riktig svar: B

NaN er en unik verdi som aldri er lik noe — heller ikke seg selv. Dette er IEEE 754-spec for floating-point. Konsekvens: du kan ikke sjekke om noe er NaN med ===.

For å sjekke om noe er NaN:

  • Number.isNaN(x) — strenge sjekk: er x nøyaktig NaN?
  • isNaN(x) — global versjon, men konverterer først: isNaN("hei") er true selv om "hei" ikke er NaN. Vanlig felle.
  • x !== x — gammel trick: bare sant hvis x er NaN.

Spørsmål 7 3 poeng Kap. 9 — Math.random spennvidde

Hva er spennvidden til verdiene som Math.random() kan returnere?

  • A Heltall mellom 0 og 100 (inklusivt begge sider).
  • B Desimaltall fra 0 (inklusivt) til 1 (eksklusivt).
  • C Desimaltall fra 0 (inklusivt) til 1 (inklusivt).
  • D Desimaltall fra -1 til 1.
Vis fasit
Riktig svar: B

Math.random() returnerer et desimaltall i intervallet [0, 1) — 0 kan forekomme, men 1 kan aldri det.

Vanlige bruksmønstre:

  • Tilfeldig heltall 0–9: Math.floor(Math.random() * 10) (gir 0, 1, …, 9 — aldri 10 fordi 1 er ekskludert).
  • Tilfeldig heltall 1–6 (terning): Math.floor(Math.random() * 6) + 1.
  • Tilfeldig element fra array: arr[Math.floor(Math.random() * arr.length)].

Hvorfor de andre er feil: A er hva man får etter Math.floor(Math.random() * 101). C overstiger spec — 1 inkluderes ikke. D er konfabulasjon.

Spørsmål 10 2 poeng Kap. 9 — Arrow function this
Ikke i pensum Arrow functions (() => {}) og this-binding er ikke gjennomgått i forelesningene. Slidesene viser kun function f() {}-deklarasjoner og objekt-konstruktører. Hopp over hvis du øver til eksamen.

Hva er den viktigste forskjellen mellom en arrow function (() => {}) og en vanlig function expression (function() {}) når det gjelder this?

  • A Ingen forskjell — this oppfører seg identisk i begge.
  • B Arrow function har sin egen this som peker til window.
  • C Arrow function arver this fra omgivende scope; vanlig function får sitt eget this basert på hvordan den kalles.
  • D Vanlig function har ingen this; arrow function alltid har en.
Vis fasit
Riktig svar: C

Arrow functions har ingen egen this — de bruker this fra den ytre scope (lexical binding). Vanlige funksjoner setter this dynamisk basert på hvordan de kalles: som metode (obj.f()) gir this = obj; som vanlig funksjon (f()) gir this = window (eller undefined i strict mode).

Praktisk konsekvens: I event-handlers er forskjellen viktig.

btn.addEventListener("click", function () {
  console.log(this);  // btn (elementet)
});
btn.addEventListener("click", () => {
  console.log(this);  // window (eller hva enn omkring var)
});

Spørsmål 11 4 poeng Kap. 9 — Array.find
Ikke i pensum Array.find er ikke nevnt i forelesningene eller læreboka 1.13.1 (som dekker filter/map/some/every/forEach). Hopp over hvis du øver til eksamen — bruk en for-løkke + early return for å oppnå samme.

Hva skriver dette ut?

let folk = [
  { navn: "Ola", alder: 22 },
  { navn: "Kari", alder: 30 },
  { navn: "Per", alder: 25 }
];
let f = folk.find(p => p.alder > 23);
console.log(f.navn);
Vis fasit
Riktig svar: Kari

Mekanisme: find() returnerer første element som passerer predikatet — ikke alle som passerer. Her iterer den gjennom Ola (22, feiler), Kari (30, passerer → returnerer Kari), og slutter. Per blir aldri sjekket.

Kontrast:

  • filter returnerer alle som passerer som ny array: [Kari, Per].
  • findIndex returnerer indeksen til første treff (1 her).
  • Hvis ingenting passerer, returnerer find undefined (ikke null, ikke tom array).

Spørsmål 12 4 poeng Kap. 9 — String.replace single

Hva skriver dette ut?

let s = "ja ja ja";

Vis fasit
Riktig svar: nei ja ja

Mekanisme: replace(streng, ny) erstatter kun det første treffet. Bare "ja" på indeks 0 byttes til "nei" — resten beholdes.

Spørsmål 15 3 poeng Kap. 9 — Undefined aritmetikk

Hva skriver dette ut?

let x;
console.log(x + 1);
Vis fasit
Riktig svar: NaN

Mekanisme: x er deklarert men ikke tildelt verdi → undefined. Ved aritmetikk konverteres undefined til NaN (Not-a-Number). NaN + 1 = NaN — alle aritmetiske operasjoner med NaN gir NaN.

Kontrast med null: null + 1 gir 1 — fordi null konverteres til 0 ved aritmetikk. Inkonsistent? Ja, det er en av JavaScripts mest bisarre kvaliteter.

Andre eksempler:

  • undefined + "a" → "undefineda" (string-konkat med konvertering)
  • undefined > 5 → false (NaN-sammenligning)
  • undefined == null → true (særskilt regel for ==)

Spørsmål 16 4 poeng Kap. 9 — Funksjons-uttrykk vs deklarasjon
Ikke i pensum Function expressions (const f = function() {}) og hoisting-forskjellen er ikke gjennomgått i forelesningene — slidesene viser kun function declarations (function f() {}) og objekt-konstruktører. Hopp over hvis du øver til eksamen.

Forklar forskjellen mellom function declaration og function expression i JavaScript. Inkluder: (a) syntaks-eksempel for hver, (b) hvordan hoisting fungerer ulikt for de to, og (c) når man typisk velger den ene fremfor den andre.

Vis fasit

Rubrikk (4 poeng):

  • 1 poeng — Function declaration: function f() { ... }. Brukes som setning, må ha navn. Hoistes med definisjon — kan kalles før den er skrevet i kildekoden.
  • 1 poeng — Function expression: const f = function() { ... } eller arrow const f = () => { ... }. Funksjonen er en verdi som tildeles en variabel. Hoistes som vanlig variabel — navnet er kjent fra toppen av scope, men selve funksjonen er undefined til linjen kjøres.
  • 1 poeng — Hoisting-forskjellen: En declaration kan kalles ovenfor sin definisjon:
    f(); // virker
    function f() { console.log("a"); }
    En expression kan ikke:
    g(); // TypeError: g is not a function
    const g = function() { ... }
  • 1 poeng — Når velge hva: Declaration når det er en gjenbrukbar, navngitt hjelper man kaller fra mange steder. Expression når funksjonen er en verdi: callback til addEventListener, IIFE, eller når man vil ha kontroll på hoisting. Arrow functions for korte uttrykk og når man trenger lexical this.

Du designer et nettsted med to skjemaer: (1) et søke-skjema der brukeren skriver "kake" og får en liste med kakeoppskrifter, og (2) et oppskrift-opplastings-skjema der brukeren legger til en ny oppskrift. Spørsmål 18 3 poeng Kap. 9 — parseInt-fellen

En student skriver følgende kode for å summere to verdier fra inputfelt:

let a = document.getElementById("a").value;
let b = document.getElementById("b").value;
let sum = a + b;
alert(sum);

Når brukeren skriver "5" og "9", får hen output "59" i stedet for 14. Forklar hvorfor, og gi to mulige rettelser i koden.

Vis fasit

Rubrikk (3 poeng):

  • 1 poeng — Diagnose: .value på et input returnerer alltid en string, uansett hvilken type input-feltet har. "5" + "9" konkateneres til "59" (siden + med strenger er sammenslåing, ikke aritmetikk). Først hvis begge operanden er tall vil + være matematisk.
  • 1 poeng — Rettelse 1: Bruk parseInt eller parseFloat for å konvertere:
    let sum = parseInt(a, 10) + parseInt(b, 10);
  • 1 poeng — Rettelse 2 (andre måter): Unary plus: let sum = +a + +b;. Eller bruk Number(): let sum = Number(a) + Number(b);. Bonus å nevne type="number" på inputet — det validerer bare, returnerer fortsatt string.

Bonus: Hvis input ikke er tall, returnerer parseInt NaN — det er greit å sjekke med isNaN() før operasjonen.

Spørsmål 19 3 poeng Kap. 8 — Klient-side vs server-side validering

Forklar forskjellen mellom klient-side validering (i nettleseren med JS/HTML) og server-side validering (i backend). Hvorfor må man alltid validere på serveren, selv om man allerede har gjort det i nettleseren?

Spørsmål 20 4 poeng Kap. 10 — addEventListener vs onclick (videre)

Forklar de fire viktigste fordelene ved å bruke element.addEventListener("click", fn) i stedet for å sette element.onclick = fn eller inline onclick="fn()" i HTML.

Vis fasit

Rubrikk (4 poeng — 1 poeng per godt punkt):

  • Flere handlere på samme event: addEventListener kan registrere mange callbacks for samme event-type. Med element.onclick = fn overskriver hver tildeling den forrige — kun én handler igjen.
  • Capturing-fasen tilgjengelig: addEventListener(type, fn, true) registrerer handleren i capturing-fasen (fra window ned mot target). onclick har bare bubbling.
  • Lett å fjerne: removeEventListener tar samme funksjons-referanse og avregistrerer. Med onclick må man manuelt sette element.onclick = null.
  • Separasjon av structure og behavior: JS i .js-filer, HTML rent. Inline onclick="..." blander de to og gjør vedlikehold vanskeligere — endrer du funksjonens navn må du oppdatere både HTML og JS.
  • Flere event-typer ut av boksen: addEventListener("scroll", ...), "keydown", "touchstart" osv. — vanlige events finnes også som on*-properties, men ikke alle.

const hoyde = document.getElementById("hoyde");
const vekt = document.getElementById("vekt");
const bmiUt = document.getElementById("bmi");
const katUt = document.getElementById("kat");

function regn() {
  const h = parseFloat(hoyde.value);
  const v = parseFloat(vekt.value);
  if (isNaN(h) || isNaN(v) || h <= 0 || v <= 0) {
    bmiUt.textContent = "–";
    katUt.textContent = "–";
    return;
  }
  const m = h / 100;
  const bmi = v / (m * m);
  bmiUt.textContent = bmi.toFixed(1);

  let kat;
  if (bmi < 18.5)      kat = "Undervekt";
  else if (bmi < 25)   kat = "Normal";
  else if (bmi < 30)   kat = "Overvekt";
  else                  kat = "Fedme";
  katUt.textContent = kat;
}

hoyde.addEventListener("input", regn);
vekt.addEventListener("input", regn);

Kommentarer:

  • input-event: Trigges hver gang verdien endres — på hvert tastetrykk eller paste. Kontrast: change trigges først når feltet mister fokus.
  • parseFloat vs parseInt: Vekt kan være 70.5 kg — vi vil ha desimal. parseInt("70.5") ville gitt 70.
  • toFixed(1): Returnerer en string med ett desimal: (22.36).toFixed(1) → "22.4" (avrundet). Ikke 22.4 som number — viktig hvis du senere skal regne videre.
  • If/else if-stigen: Sjekker fra minst til størst. Når én betingelse er sann, hoppes resten over — derfor trenger vi ikke skrive bmi >= 18.5 && bmi < 25, det andre kravet er implisitt.

Vanlige feil: Glemme å dele på 100 — får BMI for cm i stedet for meter, og resultatet er 10000x for lite. Bruke parseInt for vekt — mister desimaler. Glemme NaN-sjekk — viser "NaN" som resultat når et felt er tomt.

HTML-en under har tre knapper: Start, Stopp og Reset. Skriv JavaScript som lager en stoppeklokke:

  2. (3 poeng) Klikk på Stopp pauser tellingen, men beholder verdien. Klikk på Start igjen fortsetter fra der den var.

Vis fasit

Referanseløsning:

const visning = document.getElementById("tid");
let sekunder = 0;
let intervallId = null;

function tikk() {
  sekunder++;
  visning.textContent = sekunder;
}

document.getElementById("start").addEventListener("click", function () {
  if (intervallId !== null) return;          // allerede i gang
  intervallId = setInterval(tikk, 1000);
});

document.getElementById("stopp").addEventListener("click", function () {
  if (intervallId !== null) {
    clearInterval(intervallId);
    intervallId = null;
  }
});

document.getElementById("reset").addEventListener("click", function () {
  clearInterval(intervallId);
  intervallId = null;
  sekunder = 0;
  visning.textContent = sekunder;
});

Kommentarer:

  • State-variabler ute av funksjoner: sekunder og intervallId er deklarert utenfor handler-funksjonene slik at alle tre knapper kan se og endre dem.

Vanlige feil: Glemme tilstandsjekk i Start — to klikk gir doble teller. Glemme intervallId = null i Stopp så Start blokkeres. Bruke setTimeout som rekursiv timer (kan fungere, men trenger annen state-håndtering).

HTML-en under har et spørsmål med fire radio-knapper. Skriv kode som:

  3. (2 poeng) Hvis ingen er valgt, vis "Velg et alternativ først" i oransje — ikke regn det som feil.
  4. (1 poeng) Etter sjekken, deaktiver alle radio-knapper og knappen (så brukeren ikke kan endre i etterkant).
Vis fasit

Referanseløsning:

const knapp = document.getElementById("sjekk");
const fasitP = document.getElementById("fasit");

knapp.addEventListener("click", function () {
  // finn valgt radio
  const valgt = document.querySelector("input[name='by']:checked");

  if (!valgt) {
    fasitP.style.color = "darkorange";
    fasitP.textContent = "Velg et alternativ først";
    return;
  }

  if (valgt.value === "paris") {
    fasitP.style.color = "green";
    fasitP.textContent = "Riktig!";
  } else {
    fasitP.style.color = "crimson";
    fasitP.textContent = "Feil — riktig var Paris";
  }

  // deaktiver alle radioer og knappen
  document.querySelectorAll("input[name='by']").forEach(r => r.disabled = true);
  knapp.disabled = true;
});

Kommentarer:

  • :checked-selektor: CSS-pseudoklassen :checked kombinerer godt med querySelector — den returnerer den ene radio-en som er valgt (eller null hvis ingen).

  • disabled: En boolean property på input-elementer. r.disabled = true deaktiverer; gjør elementet uklikkelig og typisk grått.
  • forEach på NodeList: querySelectorAll returnerer en NodeList som har forEach direkte. getElementsByName returnerer derimot en HTMLCollection som ikke har det — der må man bruke vanlig for-loop eller Array.from().

Vanlige feil: Iterere uten å sjekke !valgt først — får TypeError. Bruke === på valgt uten å sjekke null — krasj på edge case. Glemme å deaktivere knappen, så brukeren kan klikke flere ganger.

Skriv en JavaScript-funksjon lagPris(beloep, valuta) som:

  1. (2 poeng) Returnerer en formatert prisstreng, f.eks. "250,00 NOK" (to desimaler, komma som desimaltegn).
  2. (1 poeng) valuta er en valgfri parameter med default-verdi "NOK".
  3. (1 poeng) Funksjonen kalles fra knappen i HTML-en og resultatet vises i <span id="ut">.
  4. (1 poeng) Bruk .toFixed(2) og .replace() til å bytte punktum med komma.
Vis fasit

Referanseløsning:

function lagPris(beloep, valuta = "NOK") {
  const formatert = beloep.toFixed(2).replace(".", ",");
  return formatert + " " + valuta;
}

document.getElementById("form").addEventListener("click", function () {
  const beloep = parseFloat(document.getElementById("bel").value);
  document.getElementById("ut").textContent = lagPris(beloep);
});

Kommentarer:

  • Default-parameter: valuta = "NOK" betyr at hvis funksjonen kalles uten andre argument (eller med undefined), brukes "NOK". Med to argumenter: lagPris(99.50, "EUR") overskriver default.
  • toFixed returnerer string: Selv om input er number, kommer .toFixed(2) tilbake som string. Det er bra her — vi kan kalle .replace rett på.
  • replace med punktum: .replace(".", ",") erstatter første treff. Det er greit her fordi toFixed(2) garantert har bare ett punktum.
  • Konkatenering: Mellom-rommet inne i " " + valuta er bevisst — "250,00 NOK" leses bedre enn "250,00NOK".

Vanlige feil: Glemme parseFloat — "250".toFixed(2) krasjer (strings har ikke toFixed). Bruke parseInt — mister desimaler. Bruke replace(/\./, ",") i stedet for vanlig string — fungerer også, men trenger ikke regex her.

Spørsmål 1 3 poeng Kap. 1 — Transportlaget

Hvilken transport-protokoll i TCP/IP-modellen er forbindelsesløs — altså sender pakker uten å bygge opp en koblingen først?

  • A TCP
  • B UDP
  • C HTTP
  • D DNS
Vis fasit
Riktig svar: B

UDP (User Datagram Protocol) er forbindelsesløs — den sender datagrammer uten hand-shake og uten å garantere levering eller rekkefølge. Brukes til video-streaming, spill, DNS-oppslag — anvendelser der lav latency er viktigere enn 100 % leveranse.

Hvorfor de andre er feil: TCP er forbindelses-orientert (bygger en sesjon, garanterer ordnet leveranse). HTTP er applikasjonslag (over TCP). DNS er applikasjonslag (kjører oftest over UDP, men er ikke selv en transportprotokoll).

Spørsmål 3 3 poeng Kap. 6 — Viewport-meta

Hva er hovedhensikten med <meta name="viewport" content="width=device-width, initial-scale=1"> i <head>-en?

  • A Den setter siden til kun å virke i 1080p oppløsning.
  • B Den ber mobile nettlesere bruke skjermens faktiske bredde i CSS-piksler og ikke zoome ut til en kunstig 980 px-bredde.
  • C Den setter en standard zoom på 100 % som ikke kan endres av brukeren.
  • D Den deaktiverer alle JavaScript-kjøringer på siden for sikkerhet.
Vis fasit
Riktig svar: B

Uten viewport-meta antar mobile nettlesere historisk at sidene ble laget for desktop og rendrer dem med en virtuell bredde rundt 980 px, deretter zoomer ned. Resultatet er liten, uleselig tekst. Med width=device-width brukes telefonens faktiske bredde i CSS-piksler, og initial-scale=1 hindrer auto-zoom-out. Dette er fundamentalt for at responsive design skal fungere på mobil.

Hvorfor de andre er feil: A er konfabulasjon. C ville krevd user-scalable=no i tillegg — og er en a11y-anti-pattern. D er konfabulasjon — viewport-meta har ingenting med JS å gjøre.

Hva kjennetegner mobile-first CSS-strategi?

  • A All CSS skrives i en egen fil som heter mobile.css og lastes først.

Spørsmål 11 4 poeng Kap. 9 — Math-runding

Hva skriver dette ut?

console.log(Math.ceil(-2.3));
Vis fasit
Riktig svar: -2

Mekanisme: Math.ceil avrunder oppover på tallinjen — alltid mot positiv side. For negative tall betyr "oppover" mot null. -2.3 avrundes opp til -2 (ikke til -3).

Sammenligning som ofte forvirrer:

  • Math.ceil(-2.3) → -2 (oppover mot 0)
  • Math.floor(-2.3) → -3 (nedover, vekk fra 0)
  • Math.round(-2.3) → -2 (vanlig avrunding)
  • Math.round(-2.5) → -2 (JS runder mot positiv ved 0.5!)

Spørsmål 12 4 poeng Kap. 9 — parseFloat

Hva skriver dette ut?

console.log(parseFloat("3.14abc"));
Vis fasit
Riktig svar: 3.14

Mekanisme: parseFloat leser tegn fra venstre og fortsetter så lenge tegnet kan inngå i et flyttall (siffer, punktum, e for eksponent). Den stopper ved første ugyldige tegn (her: a) og returnerer det den har lest så langt.

Kontrast — Number(): Number("3.14abc") → NaN fordi Number krever at hele strengen er et gyldig tall — ingen "tilgir og fortsett"-oppførsel.

Andre tilfeller:

  • parseFloat(" 3.14") → 3.14 (whitespace foran tillates)
  • parseFloat("abc3.14") → NaN (ingen gyldige tegn først)
  • parseFloat("") → NaN

Spørsmål 13 3 poeng Kap. 9 — null vs undefined

Hva skriver dette ut?

console.log(null == undefined, null === undefined);
Vis fasit
Riktig svar: true false

Mekanisme:

  • null == undefined → true. Loose-equality (==) har en spesialregel: null og undefined er "like" til hverandre (men ingen andre verdier).
  • null === undefined → false. Strict-equality (===) krever samme type — null er type object (kjent JS-bug), undefined er type undefined.

Praktisk konsekvens: Vil du sjekke "verdi er enten null eller undefined", kan du skrive x == null (én sjekk dekker begge). Vil du skille dem, må du bruke ===.

Spørsmål 14 4 poeng Kap. 9 — Array.map / filter

Hva skriver dette ut? (svar med array-syntaks, f.eks. [1,2] eller [1, 2])

let tall = [1, 2, 3];
let svar = tall.map(x => x * 2).filter(x => x > 2);
console.log(svar);
Vis fasit
Riktig svar: [4, 6]

Mekanisme — chained array-metoder:

  1. tall.map(x => x * 2) dobler hvert element: [2, 4, 6].
  2. .filter(x => x > 2) beholder bare de som er strengt større enn 2: [4, 6] (verdien 2 droppes fordi 2 > 2 er falskt).

Begge metoder returnerer et nytt array — de muterer ikke det opprinnelige. Det er en av styrkene ved funksjonell array-API: du kan kjede dem trygt uten å bekymre deg for sideeffekter.

Vanlig feil: Tro at x > 2 inkluderer 2 (det er strengt større, så 2 selv er ekskludert).

Spørsmål 15 3 poeng Kap. 9 — String.repeat

Hva skriver dette ut?

console.log("ha".repeat(3) + "!");
Vis fasit
Riktig svar: hahaha!

Mekanisme: "str".repeat(n) returnerer strengen gjentatt n ganger. "ha".repeat(3) = "hahaha". Deretter konkatenerer + strengen "!" på slutten: "hahaha!".

Spørsmål 16 4 poeng Kap. 6 — Mobile-first

Forklar forskjellen mellom mobile-first og desktop-first CSS-strategi. Hvorfor anbefales mobile-first i moderne webdesign? Nevn minst to konkrete grunner.

Vis fasit

Rubrikk (4 poeng):

Spørsmål 17 3 poeng Kap. 1 — TCP/IP-lag

Beskriv kort hva hver av de fire lagene i TCP/IP-modellen gjør, og nevn én eksempel-protokoll per lag.

Vis fasit

Rubrikk (3 poeng — 0.75 per lag):

  • Application layer — kommunikasjon med applikasjoner og brukere. Eksempler: HTTP, HTTPS, SMTP, FTP, DNS.
  • Transport layer — pålitelig (eller upålitelig) endepunkt-til-endepunkt-overføring; deler opp data i pakker og styrer flyten. Eksempler: TCP (pålitelig), UDP (forbindelsesløs).
  • Network/Internet layer — adressering og ruting av pakker mellom nettverk. Eksempler: IP, ICMP.
  • Link/Physical layer — selve fysiske mediet og dataoverføringen over kabel/radio. Eksempler: Ethernet, WiFi (IEEE 802.11).

Vanlig forveksling: OSI-modellen har 7 lag; TCP/IP-modellen 4. Det er TCP/IP som er sentral i pensum her.

Spørsmål 18 3 poeng Kap. 3 — CSS-plassering

Forklar forskjellen mellom external, internal og inline CSS. Når brukes hva, og hvilken regel "vinner" hvis det er konflikt mellom dem?

Vis fasit

Rubrikk (3 poeng):

  • 1 poeng: Tre stedene definert riktig:
    • External: egen .css-fil linket via <link rel="stylesheet" href="..."> i <head>.
    • Internal: CSS-regler i en <style>-blokk i <head>.
    • Inline: style="color: red" som attributt direkte på enkeltelementer.
  • 1 poeng: Bruksområder — external er standard for produksjon (kan caches, gjenbruk på tvers av sider, separasjon av structure/presentation). Internal brukes for sidespesifikk styling. Inline brukes typisk bare for dynamisk overstyring fra JS eller spesielle unntakstilfeller.
  • 1 poeng: Specificity-rekkefølge: inline (specificity 1000) "vinner" over both internal og external — de to siste har samme specificity, så regelen som kommer sist i kilden vinner. !important overstyrer alt uansett kilde.

Bonus å nevne: Inline-CSS bryter med "separation of concerns" og er vanskelig å vedlikeholde — derfor unngås det i moderne kodebaser.

Spørsmål 19 4 poeng Kap. 5 — Flexbox vs Grid

Forklar hovedforskjellen mellom Flexbox og CSS Grid. Når foretrekker du det ene framfor det andre? Gi ett eksempel på et layout-problem som passer best for Flexbox og ett som passer best for Grid.

Vis fasit

Rubrikk (4 poeng):

  • 1 poeng: Hovedforskjell — Flexbox er én-dimensjonal (kontrollerer enten rad eller kolonne om gangen), Grid er to-dimensjonal (rader og kolonner samtidig).
  • 1 poeng: Flexbox er innholds-drevet: barna bestemmer størrelsen sin og flex-en fordeler det. Grid er layout-drevet: containeren definerer et rutenett, og barna plasseres inn i det.
  • 1 poeng: Flexbox-eksempel — en horisontal navigasjonsmeny der lenkene fordeles med justify-content; en knapperad; en form-row med label og input. Alt som er én linje med variabel innholdsbredde.
  • 1 poeng: Grid-eksempel — en hovedside-layout med header / sidebar / hovedinnhold / footer; et galleri av kort i et auto-fyllt rutenett; en kalender. Alt som har struktur i begge dimensjoner.

Bonus å nevne: De kan brukes sammen — en Grid-container kan ha et Flexbox-barn (eller omvendt). I praksis bruker man begge.

Spørsmål 20 3 poeng Kap. 2 — Semantisk HTML

Hva betyr "semantisk HTML", og hvorfor er det bedre å bruke <header>, <nav>, <main>, <article>, <footer> osv. enn å bruke <div> til alt? Nevn minst to konkrete fordeler.

Spørsmål 21 10 poeng Kap. 5 — Flexbox-layout

HTML-en under er gitt. Skriv kun CSS som lager en typisk sidelayout med Flexbox:

  1. (2 poeng) <header> og <footer> tar full bredde med blå (#3b6ea8) bakgrunn og hvit tekst, og 16px padding.
  2. (4 poeng) .midt er en flex-container der <aside> tar ~30 % av bredden (venstre side) og <main> tar resterende ~70 % (høyre side). Bruk flex-egenskapen, ikke fastsatt bredde.
  3. (2 poeng) <aside> har grå bakgrunn (#eee) og 12px padding; <main> har hvit bakgrunn og 12px padding.
  4. (2 poeng) Hele body skal ha minst 100vh høyde, og footer skal "klistre" til bunnen selv om innholdet er kort. (Tips: gjør body til flex-column.)
Vis fasit

Referanseløsning:

* { box-sizing: border-box; margin: 0; }

body {
  display: flex;
  flex-direction: column;
  min-height: 100vh;
}

header, footer {
  background: #3b6ea8;
  color: white;
  padding: 16px;
}

.midt {
  display: flex;
  flex: 1;          /* tar all gjenstående høyde mellom header og footer */
}

aside {
  flex: 0 0 30%;    /* ingen voks, ingen krympe, basis 30% */
  background: #eee;
  padding: 12px;
}

main {
  flex: 1;          /* tar resten */
  background: white;
  padding: 12px;
}

Kommentarer:

  • body som flex-column + min-height: 100vh: Gjør at hele body fyller skjermen vertikalt, og .midt med flex: 1 tar all overflødig høyde — så footer alltid synker til bunnen ("sticky footer"-mønsteret).
  • flex: 0 0 30% vs width: 30%: Begge fungerer i mange tilfeller. flex: 0 0 30% er mer eksplisitt: "ikke voks, ikke krymp, start på 30 % bredde". Det er det rette i flexbox-kontekst.
  • flex: 1 på main: Stenografi for flex: 1 1 0% — vokse fritt, krympe fritt, basis 0. Resultat: tar resten av plassen.
  • box-sizing: border-box: Sørger for at padding ikke legger seg utenpå flex-basis-bredden og brekker layouten.

Vanlige feil: Bruke width: 30% uten flex-shrink: 0 — kan krympes uventet. Glemme min-height: 100vh så footer ikke synker. Bruke float: left — det er en pre-flexbox-teknikk som er erstattet av Flexbox/Grid for layout.

Spørsmål 22 8 poeng Kap. 6 — Responsiv navigasjon

HTML-en under er gitt: en navigasjonsmeny som default er vertikal (mobil-first). Skriv CSS som:

  1. (3 poeng) Default — på mobil: lenkene stables vertikalt, hver lenke har 10px padding, ingen list-style-bullet.
  2. (4 poeng) På skjermer som er 600 px eller bredere (bruk en media query med min-width): lenkene ligger horisontalt med 20px mellom seg (bruk Flexbox eller justify-content).
  3. (1 poeng) Lenkenes farge: mørk grå default, rød ved hover.
Vis fasit

Referanseløsning:

/* Default — mobil */
.meny {
  list-style: none;
  padding: 0;
  margin: 0;
}
.meny li a {
  display: block;
  padding: 10px;
  color: #333;
  text-decoration: none;
}
.meny li a:hover {
  color: red;
}

/* Desktop — 600px og oppover */
@media (min-width: 600px) {
  .meny {
    display: flex;
    gap: 20px;
  }
}

Kommentarer:

  • Mobile-first prinsipp: Den vertikale stylingen står utenfor noen media query — den er default for alle skjermer. Media-querien legger på horisontal flex bare når viewport er stor nok.
  • display: block på lenkene: Gjør at hele padding-arealet blir klikkbart, ikke bare teksten. Bedre touch-target på mobil.
  • gap: 20px: Moderne måte å mellom-romme flex-barn. Alternativet er .meny li:not(:last-child) { margin-right: 20px; }, men det er mer skriving.
  • Test responsiviteten: Endre vindusbredde og se overgangen ved 600 px.

Vanlige feil: Skrive desktop-CSS som default og bruke max-width: 599px for mobil (desktop-first — fungerer, men ikke mobile-first). Glemme list-style: none så punktene henger igjen. Sette padding på li i stedet for a — da er det bare teksten som er klikkbar.

Merk: I sandbox-iframen er det ikke garantert at videoen kan spilles av (autoplay-policy, format-støtte). Fokuser på at koden er korrekt strukturert — logikken er det viktige, ikke at du ser bevegelse.

Vis fasit

Referanseløsning:

const film = document.getElementById("film");
const playBtn = document.getElementById("play");
const pauseBtn = document.getElementById("pause");

playBtn.addEventListener("click", function () {
  film.play();
});

pauseBtn.addEventListener("click", function () {
  film.pause();
});

film.addEventListener("play", function () {
  playBtn.disabled = true;
  pauseBtn.disabled = false;
});

film.addEventListener("pause", function () {
  playBtn.disabled = false;
  pauseBtn.disabled = true;
});

// Initialiser: før noe spiller, pause-knappen er deaktivert
pauseBtn.disabled = true;

Kommentarer:

  • Video som DOM-objekt: <video>-elementet har et rikt API: .play(), .pause(), .currentTime, .duration, .volume, og events som play, pause, ended, timeupdate.
  • Hvorfor lytte på video, ikke på knappene: Hvis vi bare bytter disabled ved klikk på knappen, fungerer det ikke hvis videoen pauses av en annen grunn (ferdig, feil, brukeren skiftet fane). Ved å lytte på selve videoen er UI-en alltid synkronisert med videoens reelle tilstand.
  • Autoplay og muted: HTML-en har muted — det er ofte nødvendig for at play() skal lykkes uten brukerinteraksjon i moderne nettlesere. Uten muted kan promise fra .play() avvises.

Vanlige feil: Endre disabled bare i klikk-handleren, ikke i event-handleren — så UI-en kan komme ut av synk. Glemme initial-tilstand så begge knappene er aktive før noe spilles. Bruke onclick-attribute inline i HTML — fungerer, men oppgaven antyder addEventListener-mønsteret.

Spørsmål 24 8 poeng Kap. 9 — Date + funksjoner

Skriv en JavaScript-funksjon dagerTilJul() som regner ut antall dager til 24. desember i inneværende år (eller neste år hvis 24. desember allerede er passert). Vis resultatet i en alert sammen med en sjekk på om dette antallet er et primtall:

  1. (3 poeng) Bruk new Date() for å hente nåværende dato, og lag en ny Date for 24. desember. Hvis 24. desember er passert, lag for neste år.
  2. (2 poeng) Konverter forskjellen til hele dager: del på 1000 * 60 * 60 * 24 og avrund med Math.ceil.
  3. (2 poeng) Skriv en hjelpefunksjon erPrimtall(n) som returnerer true hvis n er primtall (≥ 2, kun delelig på 1 og seg selv).
  4. (1 poeng) Vis alert med begge deler, f.eks. "42 dager til jul — 42 er ikke primtall." Bind funksjonen til knappens click-event.
Vis fasit

Referanseløsning:

function erPrimtall(n) {
  if (n < 2) return false;
  for (let i = 2; i * i <= n; i++) {
    if (n % i === 0) return false;
  }
  return true;
}

function dagerTilJul() {
  const naa = new Date();
  let jul = new Date(naa.getFullYear(), 11, 24);  // måned 11 = desember
  if (naa > jul) {
    jul = new Date(naa.getFullYear() + 1, 11, 24);
  }
  const ms = jul - naa;
  const dager = Math.ceil(ms / (1000 * 60 * 60 * 24));
  const tekst = erPrimtall(dager)
    ? dager + " dager til jul — " + dager + " er primtall."
    : dager + " dager til jul — " + dager + " er ikke primtall.";
  alert(tekst);
}

document.getElementById("sjekk").addEventListener("click", dagerTilJul);

Kommentarer:

  • Måneds-indeks: JS bruker 0-indekserte måneder, så desember = 11 (ikke 12). Klassisk feil.
  • Sammenligning av Date: naa > jul sammenligner via implisitt valueOf() til millisekunder siden epoch — virker som forventet.
  • Primtall-sjekk: Optimaliseringen i * i <= n i stedet for i <= n sparer mye tid på store n (vi trenger bare sjekke opp til kvadratroten). For pedagogiske formål er i <= Math.sqrt(n) like korrekt.
  • Avrunding: Math.ceil sørger for at hvis det er 0.3 dager til midnatt (under et døgn), får vi "1 dag" — ikke 0. For "X dager igjen"-formuleringen er det vanligvis ønsket.

Vanlige feil: Bruke måned 12 (= januar neste år, off-by-one). Glemme å håndtere det at jul kan være passert. Skrive primtall-sjekken med i < n / 2 — fungerer, men ineffektivt. Skrive en versjon som kraker på n = 1 eller n = 0.

Spørsmål 25 6 poeng Kap. 5+6 — Responsivt kort-grid

HTML-en under inneholder seks "kort". Skriv CSS som lager et responsivt grid av disse uten media queries — la antall kolonner justere seg automatisk basert på tilgjengelig bredde:

  1. (3 poeng) Bruk grid-template-columns: repeat(auto-fit, minmax(200px, 1fr)) slik at hver kort er minst 200 px bredt, og det blir så mange kolonner som plassen tillater.
  2. (2 poeng) 16 px mellomrom mellom kortene (gap).
  3. (1 poeng) Hvert kort har lys grå bakgrunn (#f4f4f4), 12 px padding og 4 px border-radius.
Vis fasit

Referanseløsning:

.grid {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(200px, 1fr));
  gap: 16px;
}

.kort {
  background: #f4f4f4;
  padding: 12px;
  border-radius: 4px;
}

Kommentarer:

  • auto-fit vs auto-fill: Begge fyller raden med så mange kolonner som passer. Forskjellen: hvis det er færre elementer enn maks-antall kolonner, gjør auto-fit at de eksisterende elementene strekker seg ut til å fylle raden, mens auto-fill beholder dem på 200 px og lar tomme spor stå igjen. auto-fit er oftest mer naturlig.
  • minmax(200px, 1fr): Hver kolonne er minst 200 px bredt, men kan vokse til 1fr av tilgjengelig plass. Resultat: smale skjermer får én eller to kolonner, brede skjermer får 3-6 kolonner — automatisk uten media queries.
  • Hvorfor dette mønsteret er populært: En enkelt linje CSS lager et grid som responder elegant på alle skjermstørrelser. Mye enklere enn en serie av @media-blokker.

Vanlige feil: Bruke repeat(3, 1fr) — gir alltid 3 kolonner uansett bredde, ikke responsivt. Glemme minmax og skrive bare auto-fit — gyldig syntaks, men kolonnene kan bli ekstremt smale uten en minimums-bredde. Bruke justify-content: space-between i stedet for gap — kan fungere, men gap er den idiomatiske måten i grid/flex.

Pensum: Kap. 5 — Grid, Kap. 6 — Responsivt design

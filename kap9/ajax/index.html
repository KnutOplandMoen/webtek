01 · Pensum 1.12

Hvorfor Ajax?

Tradisjonelt sendte nettleseren en ny request hver gang du klikket på en lenke eller submitterte et skjema. Hele siden ble lastet på nytt — selv om bare en liten del trengte å oppdateres.

Ajax (Asynchronous JavaScript and XML) er teknikken der JavaScript sender en HTTP-request i bakgrunnen og oppdaterer bare den biten av siden som trenger ny data. Brukeren ser ingen «full reload». Dette er det som lar deg trykke «like» på et innlegg uten at hele Facebook flikker, eller skrive en chat-melding uten at meldings­listen hoppes til toppen.

Verktøyet pensum bruker er XMLHttpRequest — det innebygde JS-API-et for å sende HTTP-requests. (Moderne kode bruker også fetch, men XMLHttpRequest er det pensum vil at du kan.)

Kjerneidé

Statisk side = en HTML-fil per request. Dynamisk side = JS oppdaterer DOM-en med data hentet i bakgrunnen via Ajax.

02 · Pensum 1.12

XMLHttpRequest — steg for steg

En typisk Ajax-call består av fire steg:

// 1. Lag en ny request-instans
const request = new XMLHttpRequest();

// 2. Fortell hva slags request det er
request.open('GET', '/newsfeed', true);

// 3. Bestem hva som skjer når svaret kommer
request.onload = function() {
  if (request.status === 200) {
    console.log(request.responseText);
  } else {
    console.log("Feilet: " + request.status);
  }
};

// 4. Send requesten
request.send();

Argumentene til open()

  1. HTTP-metode — 'GET', 'POST', 'PUT' eller 'DELETE'.
  2. URL — pekeren til ressursen.
  3. Asynkron-flagg — true = asynkront (anbefalt). false = synkront, blokkerer hele UI-en til svaret kommer.

Hva får du tilbake

request.status
HTTP-statuskoden — 200, 404, 500 osv.
request.responseText
Selve dataen tjeneren sendte, som tekststreng.
request.onload
Funksjonen som kjøres når svaret er klart.

Hvis serveren svarte med JSON, parser du teksten:

request.onload = function() {
  if (request.status === 200) {
    const data = JSON.parse(request.responseText);
    console.log(data.tittel);
  }
};
MCQ · 1.12
I request.open('GET', url, true), hva betyr det tredje argumentet (true)?
  • At cache skal brukes
  • At requesten skal være asynkron
  • At svaret skal parses som JSON
  • At cookies skal sendes med
Async-flagget. true = ikke-blokkerende (varsel via onload). false = blokkerer UI-en til svaret kommer.
MCQ · 1.12
Hvilken egenskap inneholder selve dataen tjeneren returnerte?
  • request.status
  • request.responseText
  • request.data
  • request.body
responseText = data som streng. status = bare statuskoden (200, 404, …).
03 · Pensum 1.12

HTTP-metoder

Det første argumentet til open() bestemmer hva du vil gjøre med ressursen:

GET
Hent data. Brukes til å lese, ikke til å endre. Ingen body.
POST
Opprett ny ressurs. Data sendes i body — typisk skjema-data eller JSON.
PUT
Oppdater eksisterende ressurs.
DELETE
Slett ressurs.
Konvensjon vs hard regel

Tjeneren kan teknisk svare på en GET som om det var en POST — men konvensjonen er at GET skal være «trygg» og uten sideeffekter. Følger du konvensjonen vet andre utviklere hva koden din gjør.

MCQ · 1.12
Hvilken HTTP-metode brukes typisk når man henter data?
  • GET
  • POST
  • PUT
  • DELETE
GET = lese. POST = opprette. PUT = oppdatere. DELETE = slette.
MCQ · 1.12
Du skal sende inn et nytt blogg-innlegg. Hvilken metode passer best?
  • GET
  • POST
  • PUT
  • DELETE
POST brukes til å opprette ny data. PUT brukes hvis du oppdaterer et innlegg som finnes fra før.
04 · Pensum 1.12

Statuskoder

Statuskoden forteller hvordan det gikk. Pensum vil at du kan disse:

200 OK
Alt gikk bra — du kan lese responseText.
400 Bad Request
Klienten sendte noe ugyldig — sjekk syntaksen i requesten.
401 Unauthorized
Du må autentisere deg for å få tilgang.
403 Forbidden
Du er identifisert, men har ikke lov.
404 Not Found
Tjeneren fant ikke ressursen.
500 Internal Server Error
Tjeneren krasjet — ikke din feil.
502 Bad Gateway
En proxy/gateway fikk ugyldig svar fra en annen server lengre bak.
503 Service Unavailable
Tjeneren er midlertidig nede eller overbelastet — prøv igjen senere.

Pensum lenker til MDNs fulle liste for alle statuskoder.

Lær gruppene

2xx = suksess. 3xx = redirect. 4xx = klientens feil. 5xx = tjenerens feil. Hvis du husker gruppen, kan du resonnere deg fram til hva en spesifikk kode betyr.

MCQ · 1.12
Hva betyr statuskoden 404?
  • OK
  • Bad Request
  • Not Found
  • Internal Server Error
404 = tjeneren fant ikke ressursen. 500 = tjener-feil. 200 = OK.
MCQ · 1.12
Hvilken gruppe statuskoder indikerer en feil hos klienten?
  • 2xx
  • 3xx
  • 4xx
  • 5xx
4xx = klientens feil (404, 401, 403). 5xx = tjenerens feil (500). 2xx = suksess.
MCQ · 1.12
Du sender en POST og får tilbake 500. Hvem skal du klage til?
  • Sjekk koden din — det er din feil
  • Tjeneren krasjet — det er på serversiden
  • Ressursen finnes ikke
  • Du er ikke autentisert
5xx er tjenerens feil. Med 500 har serveren krasjet på et vis den ikke vet hvordan håndteres.
05 · Pensum 1.12

Same-origin policy og JSONP

Av sikkerhetshensyn lar nettleseren som standard ikke et script på nrk.no sende en XMLHttpRequest til vg.no. Det kalles same-origin policy: requesten må gå til samme «origin» (samme protokoll, host og port).

Hadde policy-en ikke vært der, kunne et ondsinnet script på en tilfeldig side ha lest cookien din fra nettbank.no.

JSONP — en gammel omveier

Mens XMLHttpRequest blokkeres på tvers av domener, gjør ikke <script src="…"> det. Du kan altså laste JavaScript fra hvilket som helst domene. JSONP (JSON with Padding) utnytter dette: tjeneren returnerer JSON pakket inn i et JavaScript-funksjonskall som scriptet ditt allerede har definert.

Moderne kode bruker CORS (Cross-Origin Resource Sharing) i stedet — det er en mekanisme der serveren eksplisitt tillater cross-origin-requests via spesielle headers. Pensum nevner JSONP som tidligere løsning.

MCQ · 1.12
Hva er Same-origin policy?
  • En regel for at HTML-filer må ha samme navn som CSS-filer
  • En sikkerhetsbegrensning som hindrer JS-requests på tvers av domener
  • En regel om at alle requests må være GET
  • En policy fra serveren om hvilke språk som er tillatt
Same-origin policy hindrer scripts fra ett domene i å sende XMLHttpRequest til andre domener — en sentral sikkerhetsmekanisme i nettleseren.
MCQ · 1.12
Hva er hovedinnsikten bak JSONP?
  • JSON er raskere enn XML
  • <script src> blokkeres ikke av same-origin policy
  • Tjeneren kan oppgi sitt eget passord
  • JSONP er kryptert
JSONP utnytter at <script>-tags kan hente JS fra andre domener — det stoppes ikke av same-origin slik XMLHttpRequest gjør.
06 · Vanlige feller

Vanlige feller

Pass på
  • Glemme send(): requesten avfyres faktisk når du kaller request.send(). Uten den skjer ingenting.
  • Synkron request (false): blokkerer hele UI-en til svaret kommer. Bruk true og onload-callback.
  • Lese responseText før onload: dataen er ikke ankommet ennå — du får tom streng eller eldre data.
  • Glemme å sjekke request.status: hvis svaret er 404 eller 500, er responseText sannsynligvis ubrukelig.
  • Cross-origin: request mot annet domene blokkeres med mindre serveren har CORS-headers. Du får en feil i konsollen, ikke et svar.
  • Glemme å parse JSON: responseText er en streng selv om innholdet er JSON. Du må kjøre JSON.parse() selv.

MCQ · Helhet
Hva er poenget med Ajax?
  • Å kryptere data i transit
  • Å hente data fra server uten å laste hele siden på nytt
  • Å erstatte HTML med JavaScript
  • Å kjøre kode på serveren
Ajax = asynkrone requests fra JS i bakgrunnen. Siden kan oppdatere deler av seg selv uten en full reload.
MCQ · Helhet
Hva er den riktige rekkefølgen for å sende en GET-request?
  • send() → open() → onload
  • new XMLHttpRequest() → open() → onload = … → send()
  • open() → send() → new XMLHttpRequest()
  • onload = … → send() → open()
Lag instans, kall open(), sett onload-handler, kall til slutt send().
MCQ · Helhet
Skriv en funksjon hentNyheter() som henter /news og logger svaret. Hva må være med i kjernen?
  • fetch('/news')
  • new XMLHttpRequest(), open('GET', '/news', true), onload, send()
  • document.querySelector('/news')
  • JSON.parse('/news')
Pensum bruker XMLHttpRequest — selv om fetch er moderne praksis, er det den gamle som testes i kapittelet.

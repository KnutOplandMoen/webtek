01 · Pensum 1.11.1

Hvor skriver du JavaScript?

Det er to måter å plassere JS i et HTML-dokument: inline eller eksternt. Begge varianter bruker en <script>-tag, og begge kan stå i <head> eller i <body>.

Inline (embedded)

Koden står direkte inni <script>-tagen:

<script>
  console.log("Hei fra scriptet");
</script>

Eksternt

Koden ligger i en egen .js-fil, og HTML-en peker til den med src-attributten: 

<script src="path/to/script.js"></script>

Når koden er ekstern slipper du å gjenta den på hver side — flere HTML-filer kan dele samme .js-fil. Det er samme DRY-tanke som med ekstern CSS.

Kjerneidé

Inline = koden er i HTML-en. Ekstern = koden er ved siden av HTML-en. Velg ekstern når koden vokser eller skal deles av flere sider.

02 · Pensum 1.11.1

Nettleseren leser ovenfra og ned

Når nettleseren leser HTML-dokumentet, går den linje for linje fra topp til bunn. Når den treffer en <script>-tag, stopper den og kjører JavaScript-koden umiddelbart. Først når scriptet er ferdig fortsetter den med resten av HTML-en.

Det betyr at JavaScript bare kan «se» de HTML-elementene som er rendret før scriptet kjøres. Hvis et script i <head> prøver å hente et <p>-element fra <body>, finnes elementet ikke ennå.

«You can't yell at a baby before it is born.»

Mental modell

Tenk på HTML som en oppskrift som leses ovenfra. JS er notater i margen — de gjelder bare det som er bakt så langt. Du kan ikke notere om kaken før den er i ovnen.

03 · Pensum 1.11.1

Tre måter å unngå fødselsproblemet

Hvis scriptet ditt skal røre HTML-elementer, har du tre standardvalg:

  1. Legg <script> nederst i <body>. Når scriptet kjøres er hele DOM-en allerede rendret. Enkelt og pålitelig.
  2. Pakk inn i window.onload. Funksjonen kjøres først når hele siden (inkludert bilder) er ferdig lastet. Da spiller plasseringen av scriptet ingen rolle. 
    window.onload = function() {
  // Kjøres etter at DOM-en er ferdig
  document.querySelector("h1").style.color = "red";
};
  3. Skill ut i en ekstern fil. Filen lastes etter at HTML-en er parset hvis du legger <script src="…"> nederst i <body>.
MCQ · 1.11.1
Hva skjer hvis et script i <head> prøver å hente et element som først er rendret nederst i <body>?
  • Nettleseren venter automatisk til elementet finnes
  • Scriptet får null tilbake — elementet finnes ikke ennå
  • HTML-rendring stopper for alltid
  • Scriptet kjøres på nytt automatisk
DOM-en bygges ovenfra og ned. Når scriptet i <head> kjøres, er <body> ikke parset enda — så elementet eksisterer ikke. Løsning: flytt scriptet, bruk window.onload, eller bruk ekstern fil.
MCQ · 1.11.1
Hva er hovedfordelen med å lagre JS i en egen fil?
  • Filen kjøres raskere
  • Flere HTML-sider kan dele samme kode (DRY)
  • Eksterne filer er nødvendig for at JS skal virke
  • Eksterne filer trenger ikke <script>-tag
DRY — Don't Repeat Yourself. Det blir også lettere å vedlikeholde, og editorer kan gi bedre syntax-highlighting og hint på rene .js-filer.
MCQ · 1.11.1
Hva gjør window.onload = function() { … };?
  • Sletter alt på siden
  • Kjører funksjonen først når hele DOM-en er rendret
  • Kjører funksjonen før HTML-en er parset
  • Lager en ny side i nettleseren
window.onload er en hendelse som fyrer når dokumentet er ferdig lastet. Da er alle elementer trygt tilgjengelige.
04 · Pensum 1.11.2

Kommentarer i JavaScript

JavaScript har to kommentar-syntakser:

  • Linjekommentar: alt etter // til linjeslutt ignoreres.
  • Blokkommentar: alt mellom /* og */ ignoreres — kan strekke seg over flere linjer.
// Dette er en linjekommentar
let a = 5; // kan også stå på slutten av en linje

/* Dette er en blokk-kommentar.
   Den kan dekke flere linjer
   uten å gjentas. */
let b = 10;

Bruk dem til to ting:

  1. Forklare hvorfor — særlig hvis koden gjør noe som ikke er åpenbart for en leser.
  2. Skru av kode midlertidig — når du eksperimenterer eller debugger.
Pass på

I motsetning til CSS (som bare har /* … */) og HTML (som bare har <!-- … -->), har JavaScript begge to. Men ikke bland kommentar-syntakser på tvers av språk — // i en CSS-fil er ikke en kommentar, det er en feil.

MCQ · 1.11.2
Hvilken syntaks er ikke en gyldig JavaScript-kommentar?
  • // kommentar
  • /* kommentar */
  • <!-- kommentar -->
  • /* multi-
    linje */
<!-- … --> er HTML-syntaks. JavaScript bruker // og /* … */.
MCQ · 1.11.2
Hva skjer med koden mellom /* og */?
  • Den kjøres bare i utviklerverktøy
  • Den ignoreres helt av JavaScript-tolkeren
  • Den kjøres dobbelt
  • Den loggføres til konsollen
Tolkeren hopper over kommentarer. Det er derfor du kan «kommentere ut» kode for å skru den av uten å slette den.
05 · Vanlige feller

Vanlige feller

Pass på
  • Script i <head> uten window.onload: elementet eksisterer ikke ennå når scriptet kjøres.
  • Glemmer </script>: resten av HTML-en blir tolket som JavaScript fram til endetagen kommer.
  • Bruker både src og inline-kode i samme tag: innholdet ignoreres når src er satt — kun den eksterne filen kjøres.
  • Glemmer å lukke en /* …-blokk: resten av filen blir kommentar.

Hvor i HTML-strukturen plasseres et script tradisjonelt for å unngå at det refererer udeklarerte elementer?
  • Øverst i <head>
  • Inni <title>
  • Nederst i <body>

Når scriptet ligger sist, er hele DOM-en parset og elementene finnes. Alternativet er window.onload.

Hvilket kodeutklipp er syntaktisk gyldig JavaScript?
  • # Dette er en kommentar
  • // Dette er en kommentar
  • -- Dette er en kommentar
  • REM Dette er en kommentar
// for én linje, /* */ for blokk. #, -- og REM hører til andre språk (Python/Bash, SQL, BASIC).
